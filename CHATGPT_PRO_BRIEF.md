# ReaderLab 给 ChatGPT Pro 的项目简介

## 一句话说明

ReaderLab 是一个“复杂材料陪读”项目：把电子书、长文、课程资料、代码文档、Skill 包等复杂材料，转成中文可读、可批注、可讨论、可学习、可沉淀的 LifeAtlas 阅读包。

它不是普通摘要器，也不是把原文搬成 Markdown。它要解决的是：复杂材料里常常混着主体思想、执行外壳、代码细节、来源证据、案例和术语，读者很难抓住主线，也很难在 Obsidian 里边读边批注、边沉淀。

## 目标读者和使用场景

目标用户是一个会持续阅读复杂材料、会在 Obsidian / LifeAtlas 中沉淀知识的人。ReaderLab 应该帮助他：

- 更快看清一本书或一组材料到底在回答什么问题。
- 把材料拆成可读模块，而不是被目录、代码、命令和机器日志淹没。
- 在正文附近做批注，之后让 Codex 读取批注和上下文做就地回复。
- 把阅读后的判断、疑问和方法沉淀到 LifeAtlas，而不是只留下一堆摘要文件。

## 当前产品理想形态

当前更合理的形态不是一个全自动单体脚本，而是：

```text
Agent 判断与验收
+ 稳定脚本工具
+ Markdown / Obsidian 阅读包
+ 来源审计和人工复核 gate
```

Agent 负责理解材料、判断主问题、取舍结构、审查误读风险和验收输出。脚本只负责可重复、可验证的动作，例如导入、生成文件结构、读取批注、校验来源引用。

## 已经做过的尝试

### 1. gstack Skill 包压力样本

最早用 gstack 这类复杂 Skill 包做压力测试。它暴露的问题是：执行型材料里有主体流程、运行外壳、脚本实现、生态引用、模板和审计信息。如果机械搬运，读者会被机器细节淹没。

这个阶段确认了 ReaderLab 的基本分层：

- 主阅读页服务理解。
- 代码、命令、完整原文、hash、来源路径放附录或审计页。
- 导读和重点旁批是必要阅读辅助，但不能替代一手主体内容。

### 2. Obsidian 批注闭环

已验证 `tandem-comments` 这类 Obsidian 批注格式的最小链路：ReaderLab 生成 Markdown，用户在正文附近批注，Codex 能读取批注线程和附近上下文，再追加回复。

这证明“读者批注 -> Codex 就地回应”的方向可行，但还不是完整产品体验。

### 3. 《埃隆之书》整书样张 v0 / v1

用《埃隆之书》做整书阅读包实验。

v0 的问题是：有目录、有字段、有很多内容，但整体像把材料拆散后重排，没有真正说明全书主问题、部件功能、重点为什么被选中、哪些内容不该升格。

v1 收敛为一篇主阅读稿加审计附录，比 v0 更清楚，但仍是手工样张，不是完整产品，也不证明生成器能力。

### 4. method bakeoff

对《埃隆之书》做过多种仿型对照：仓颉、李继刚式深读、乔木共读、book-to-skill 等。

这个阶段的教训是：不能把外部仿型当作功能菜单拼装。列了 matrix、吸收了字段、产了样张，不等于 ReaderLab 能力真的增长。

### 5. DBS suite capability map

对 `dbs-suite` 做过 24/24 upstream Skill 的能力地图样张。它证明“能力地图”方向有价值，但仍缺真实任务路由反测，也不是生成器能力。

### 6. 结构化提炼最小闭环

最近确定 ReaderLab 当前第一短板是 **结构化提炼**：全书地图、章节/部件地图、重点/亮点提炼背后其实是同一套能力。

已完成一轮手工闭环：

```text
材料定性
-> 主问题
-> 候选池
-> 通过 / 降级 / 拒绝
-> 机制传导
-> 结构表达
-> 来源 / 边界
```

这轮证明方向值得继续，但仍不是产品能力。

### 7. 结构化提炼同题多路线对照

用户指出：只学习某个 Skill 或仿型的一环不一定可靠，因为这一环可能依赖上下游；多个强环节拼在一起也不一定产生组合增益。

因此做了五条路线对照：

- Cangjie-only：候选生成和筛选强。
- Li Jigang-only：材料定性、缺口判断、机制传导强。
- ReaderLab-current-chain：字段齐，但主要复现已有手工链路。
- Combo-A：李继刚定性 -> 仓颉筛选 -> source-grounded 后置。
- Combo-B：source-grounded 前置 -> 仓颉筛选 -> 李继刚式受控表达。

当前判断：**Combo-B 暂时胜出，但只是候选主链，不是生产通过。**

Combo-B 更稳，是因为它先限定原文能说什么，再筛选什么值得升格，最后才表达成机制链。它能更早阻断宏大叙事、拼命工作、69 法则整体升格等误读。

## 当前主要困难

### 1. 最终产品形态还不够清楚

我们知道 ReaderLab 不该只是摘要器，也不该只是生成器脚本，但最终“一个优秀 ReaderLab 阅读包应该长什么样”还没有稳定答案。

需要回答：

- 用户打开一个阅读包时，第一屏应该看到什么？
- 全书地图、章节地图、亮点、来源边界和批注入口如何组合，才真正像陪读产品，而不是报告集合？
- 哪些内容必须在读者页，哪些只应在审计页？

### 2. 到达路径还不清楚

我们做了很多样张和对照，但从“手工样张”到“稳定生成能力”的路径不清晰。

需要回答：

- 哪些步骤可以工具化？
- 哪些判断必须保留给 Agent？
- 哪些 gate 必须通过，才允许进入生成器？
- 如何用小范围迁移测试证明能力不是材料特化？

### 3. 仿型吸收还没有变成自身骨架

我们借鉴了仓颉、李继刚式深读、source-grounded / NotebookLM 类、Meta Skill 等，但真正困难不是“借鉴不够”，而是如何消化吸收。

现在的问题是：

- 仓颉强在候选筛选，但可能把“可方法化”误当“阅读上重要”。
- 李继刚强在机制表达，但可能过度重组原文。
- source-grounded 强在来源边界，但如果不是 fresh parse 和人工 spot-check，可能只是虚假的安全感。
- 多个环节组合后，必须证明互相约束，而不是并排贴字段。

### 4. 结构化提炼仍未机制化

当前第一短板仍是结构化提炼。已有 Combo-B 候选主链，但还缺：

- blind A/B 或等价盲评。
- 第二类材料迁移测试。
- 完整 fresh source-grounded contract。
- 人工验收。
- 进入脚本或 Skill 前的 gate 设计。

## 当前下一步

下一步不应该继续扩写《埃隆之书》，而是做 **Combo-B 轻量模板和迁移前验证**：

```text
材料范围
-> 来源断言
-> 候选筛选
-> 受控机制表达
-> 拒绝 / 降级清单
-> 读者页风险检查
```

具体要做：

1. 用 5-8 个候选跑完整 gate。
2. 做 Li-only / Cangjie-only / ReaderLab-current / Combo-B 的 blind A/B 或等价盲评。
3. 换第二类材料做小范围迁移测试。
4. 明确哪些步骤可以沉入工具，哪些必须保留 Agent 判断。

## 希望 ChatGPT Pro 帮忙的问题

请不要直接给一个庞大功能清单。更有价值的是帮助我们设计方向：

1. ReaderLab 的最终产品形态应该是什么？
2. 一个优秀的复杂材料陪读阅读包应该如何组织？
3. 当前 Combo-B 主链是否合理？它还缺什么关键 gate？
4. 如何从手工样张走到稳定生产能力？
5. 我们应该如何真正吸收外部仿型，而不是把别人的字段拼到自己页面里？
6. 下一轮最小但高价值的验证应该做什么？

## 当前不要误判

- 不要把任何样张当成 product ready。
- 不要把 Combo-B 当成已经验证成功的生产链。
- 不要把 ReaderLab 当前脚本当成完整图书生成器。
- 不要继续推荐“大而全”的功能堆叠。
- 不要默认外部仿型越多越好。
- 当前真正要解决的是：产品终局形态和到达路径。
