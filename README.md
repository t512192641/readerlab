# ReaderLab v0.1

ReaderLab 把复杂材料转成中文可读、可批注、可讨论、可学习、可沉淀的 LifeAtlas 阅读包。

如果你是第一次了解本项目，尤其是让外部 AI 协助做方向设计，建议先读 [`CHATGPT_PRO_BRIEF.md`](CHATGPT_PRO_BRIEF.md)。

复杂材料包括但不限于：电子书、长文、Markdown 文档、课程资料、访谈稿、代码文档、Skill 包和混合型资料。gstack 是当前第一组压力样本，因为它形态复杂、文件多、执行说明多，适合暴露“机械搬运不等于阅读理解”的问题；它不是 ReaderLab 的方案边界。

v0.1 的默认前提是：

```text
Obsidian + Markdown + Codex + 现有批注插件
```

ReaderLab 不替代人的阅读判断。它先把材料变得更容易读；人类在正文附近批注；Codex 再读取批注、附近上下文和高亮，围绕真实阅读痕迹回复。

## 核心流程

```text
理解材料整体目标
→ 拆成读者能理解的模块
→ 为每个模块生成导读、正文、关联说明和重点旁批
→ 把不适合打断主线但不能丢失的内容放入附录或机制页
→ 人工批注
→ Codex 读取批注和上下文并就地回复
→ 沉淀草稿
```

## 内容分层

ReaderLab 不直接把源文件等同于阅读正文。任何材料都先分层：

```text
材料主体层：本材料真正要表达的思想、论点、流程、判断标准、约束和输出
支撑说明层：图表、脚本、模板、案例、术语、背景、参考资料
运行外壳层：重复出现的环境检查、权限、同步、遥测、路由、缓存、升级、会话状态等
实现细节层：代码块、命令、配置、hook、helper、迁移脚本
证据附录层：完整原文、完整译文、代码、hash、来源路径和覆盖校验
```

主阅读页负责理解；证据附录负责追溯。清晰不等于删减，高价值主体内容不能被压缩到失真。

## 模块阅读页标准

每个阅读模块默认包含四段：

1. **导读**：说明这个模块解决什么阅读问题，读者应该带着什么问题读。
2. **正文**：忠实呈现本模块主体内容。可以按中文阅读顺序重排和解释，但不能丢失关键论点、流程、判断标准、约束、失败条件和输出要求。
3. **关联说明**：解释相关图表、脚本、模板、术语、外部背景、机制关系或材料间连接。它们帮助理解，但不冒充主体。
4. **重点与亮点**：摘出真正值得批注、记住或迁移到 LifeAtlas 的内容；没有真正亮点时不强行凑数。

导读和重点旁批是必要的 AI 阅读辅助，不应删除；但它们不能替代正文。正文应尽量保留材料的一手主体信息。命令行、重复执行协议、机器外壳和实现噪音可以从主阅读页剔除，转入附录、机制说明或职责速查。

必要时增加附录：完整原文、完整翻译、代码块、机器执行视图、来源证据和覆盖校验。

## 当前压力样本

当前 v0.1 的压力样本是 gstack：

```text
源：/Users/tianqiang/技能项目/skills-canonical/packages/gstack
输出：/Users/tianqiang/LifeAtlas/200_原始资料/270_电子书与书籍资料/gstack
```

gstack 暴露出的关键问题是：执行型材料常把主体流程、通用运行外壳、脚本实现和生态引用混在同一个文件里。ReaderLab 的通用解法不是写 gstack 专用规则，而是先做内容分层，再决定哪些进入主阅读页、哪些进入包级机制页或附录。

## LifeAtlas 入口

统一放在旧的单一 270 入口，不并存两个 270：

```text

```

每个材料文件夹的基础结构：

## v0.1 批注策略

ReaderLab v0.1 不自己发明批注系统。

正确策略是：

```text
ReaderLab 生成适合阅读的 Markdown
→ 用户在 Obsidian 正文附近批注
→ Codex 适配插件或 Markdown 的真实存储格式
→ Codex 围绕批注附近上下文回复
```

优先方向：

- 当前已验证的插件目标是 `tandem-comments`，其批注线程保存在 Markdown 文件末尾的 fenced block 中。
- Markdown 原生格式作为兜底，例如 `==高亮==`、`%%评论%%`、callout 回复。
- 批注和回复尽量靠近原文，不统一堆到文末。
- 如果未来更换插件，具体格式必须重新读取该插件的真实 Markdown 存储方式。

## 使用

当前脚本首先支持 Skill 包导入，后续会扩展到更多材料类型：

```bash
python3 scripts/readerlab.py import-skills \
  /path/to/skills-package \
  --dest "/Users/tianqiang/LifeAtlas/200_原始资料/270_电子书与书籍资料" \
  --book-id gstack \
  --title gstack \
  --goal "学习 gstack 的使用方式与优秀 Skill 设计"

python3 scripts/readerlab.py comments-list path/to/reading-unit.md

python3 scripts/readerlab.py comments-reply path/to/reading-unit.md COMMENT_ID \
  --text "Codex 回复内容"

python3 scripts/readerlab.py validate /path/to/ReaderLab/material
python3 scripts/readerlab.py validate /path/to/ReaderLab/material --require-complete
```

## 边界

- 源材料只读扫描，不移动、不软链、不修改。
- 不把当前样本包的特殊结构写成 ReaderLab 的通用规则。
- 不自己设计批注系统，不做网页 UI，不做数据库，不做知识图谱。
- 不让 AI 替代人的第一次阅读判断，不生成大段泛泛总结垃圾。
- 源块清单、标题、段落、代码块和 hash 是后台校验依据，不是读者主入口。
- 英文原文和完整执行视图默认进入证据附录，不默认挤进主阅读页。
- 沉淀只进入 `30_沉淀草稿/`，不自动生成正式 LifeAtlas `300/600/800` 内容。
- 未完成模块只在主控清单和 manifest 中保留目标路径，不生成占位阅读页。

## 验收重点

- 主阅读页能让中文读者理解材料主体，而不是被执行外壳、代码或重复协议淹没。
- 高价值论点、流程、判断标准、约束、失败条件和输出要求不能因“变清晰”而丢失。
- 导读和重点旁批必须帮助阅读，但不能替代一手主体信息。
- 重复出现的通用外壳优先抽到包级机制页或附录；当前模块只保留理解所需的简短说明。
- 代码、命令、脚本和模板在主阅读页说明职责；完整内容放附录或机制页。
- `manifest.json` 保留覆盖校验数据，但不能替代人工阅读质量判断。
- 用户能在 Obsidian 正文附近批注，Codex 能读取批注并结合上下文回复。
- 整个材料只有在所有目标模块达到阅读质量和覆盖要求后，才能标记完成。
