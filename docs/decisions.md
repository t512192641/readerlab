## D-014 ReaderLab 面向复杂材料陪读，不是 Skill 专项

ReaderLab 的根目标是把复杂材料转成中文可读、可批注、可讨论、可学习、可沉淀的 LifeAtlas 阅读包。复杂材料包括电子书、长文、Markdown 文档、课程资料、访谈稿、代码文档、Skill 包和混合型资料。

v0.1 的默认形态是 Obsidian + Markdown + Codex + 现有批注插件。ReaderLab 先降低阅读门槛，人类在正文附近批注，Codex 再读取批注和附近上下文并回复。

当前 gstack 只是第一组复杂压力样本，用来暴露和验证方法；它不是 ReaderLab 的方案边界。任何 SOP、校验规则或文档表达都不能把 gstack 的特殊结构写成通用规则。

后续规则必须先描述通用材料阅读原则，再说明当前实现如何在 Skill 包样本上落地。

## D-015 主阅读页和证据附录分工

ReaderLab 不再把“完整翻译源文件”直接等同于“主阅读页正文”。任何材料都必须先做内容分层：

- 材料主体层：思想、论点、流程、判断标准、约束、失败条件、输出要求。
- 支撑说明层：图表、案例、术语、脚本职责、模板关系、外部背景。
- 运行外壳层：重复出现的环境检查、权限、同步、遥测、路由、缓存、升级、会话状态。
- 实现细节层：代码块、命令、配置、hook、helper、迁移脚本。
- 证据附录层：完整原文、完整译文、代码、hash、来源路径和覆盖校验。

主阅读页负责让读者完整理解材料主体；证据附录负责防止信息流失和支持审计。清晰不等于摘要化，附录完整也不等于主阅读页必须塞入所有执行细节。

AI 生成的导读、阅读地图和必要关联说明是阅读辅助，不能替代正文。旧 `重点与亮点` 已确认下线。书籍和长文默认原样保留正文；Skill 包和工程材料默认保留净化后的主体信息，尤其是论点、流程、判断标准、约束、失败条件、输出要求和高价值表达。

命令行、重复执行协议、机器外壳和实现噪音可以从主阅读页剔除，转入附录、机制说明或职责速查。剔除主线不等于丢弃证据。

## D-016 不为单个样本包设专用通用规则

ReaderLab 可以在实现内部利用某个材料包的结构线索提高识别准确率，但项目文档和 SOP 不应为单个样本包建立特殊地位。

例如 gstack 中的 `{{PREAMBLE}}`、`preamble-tier`、生成型 `SKILL.md`、通用运行外壳等，只能作为“运行外壳层”的样本现象，不应成为 ReaderLab 通用规则的主干。

通用判断应基于内容功能：是否是材料主体、是否支撑理解、是否重复外壳、是否实现细节、是否审计证据。

## D-017 v0.1 不自建批注系统

ReaderLab v0.1 不自己发明批注系统，不做网页 UI，不做数据库，也不做知识图谱。Markdown 是数据本体，Obsidian 是阅读界面。

批注优先采用现有 Obsidian 插件；Markdown 原生格式作为稳定兜底，例如 `==高亮==`、`%%评论%%` 和 callout。具体解析与写回规则必须以插件或 Markdown 的真实存储格式为准。

批注必须靠近原文。Codex 回复也应尽量写在对应批注附近；如果插件格式不支持就地回复，才使用最接近原文位置的 Markdown callout 追加回复。

AI 不能替代人的第一次阅读判断。预处理目标是让材料更容易读，不是生成大段总结，也不是自动提炼长期原则或自动升格到正式知识库。

## D-018 阅读单元不是固定书架

ReaderLab 不能假设存在一套固定目录能装下所有复杂材料。电子书、课程、访谈、代码文档、Skill 包和混合资料的阅读单元都应先从材料自身结构和内容功能中判断。

当前“通用阅读角色分类”只是 v0.1 的过渡规则分类器，用来替代 gstack 固定组和简单关键词兜底；它不是最终产品模型，也不是所有材料必须套用的固定分组。

生成器应输出材料包级 `reading_units` 规划，至少记录每个单元的用途、判断理由、判断依据和把握程度。对 Skill 包，应先读包入口、路由关系、`SKILL.md` 正文职责以及脚本/模板职责；对图书和长文，应先读书名、前言、目录和章节结构。

“待人工分组”不能作为正常出口。机器读完整体结构仍分不清时，应输出“材料结构诊断”，说明入口缺失、职责混杂、路由矛盾或阅读价值存疑；人的职责是验收路线或决定是否继续处理，不是替机器完成基础分组。

## D-019 厚页可接受，摘要化不可接受

v0.1 demo 的首要目标是路线正确：保留一手主体信息，剔除或转移命令行、重复外壳和机器细节，并支持 Obsidian 批注闭环。

如果某个材料本身就是长库、公式集、清单库或高密度框架，主阅读页可以先保持较厚。厚页不是错误；把主体资产压成短摘要、删掉关键判断链、或为了统一模板过早拆分才是错误。

当人工阅读确认某页过厚时，再对该页做二次拆解、拆出附录或公式库。不要为了单个极端样本提前把生成器架构过拟合。

## D-020 重建阅读包必须保护批注

Obsidian 批注和 Codex 回复是 ReaderLab v0.1 的核心数据，不是可随意覆盖的派生产物。

生成器在覆盖或重建已存在的 LifeAtlas 阅读包前，必须检查目标 Markdown 中是否存在当前插件格式的批注数据。当前插件目标是 `tandem-comments`，其数据位于文件末尾 fenced block。

没有批注时可以按用户授权覆盖；存在批注时必须先保存批注数据并执行风险审核。最低要求是不能丢失批注 thread、anchor exact/prefix/suffix/pos、来源页面和对应材料/章节。是否迁移、跳过覆盖或请用户确认，应由写入策略明确处理。

## D-021 机器完成不等于人工验收

ReaderLab 的机器状态只能证明已定义的结构和覆盖规则通过，不能证明阅读质量通过。

后续状态模型应区分机器完成和人工验收：例如 `machine_completed` 表示生成器、覆盖、hash、残留检查等通过；`human_accepted` 表示用户或指定人工审阅确认该页面真的适合阅读。

在状态模型拆分前，所有 validate 通过的结论都应表述为“机器验收通过”或“可供人工审阅”，不能把 validator 绿灯说成阅读质量最终通过。

## D-022 主阅读优先，机制按需，证据退后

ReaderLab 的默认输出不应让读者在 `10_中文精读`、`11_机制说明`、`12_证据附录` 三棵镜像目录之间来回跳。三层结构可以作为内部逻辑，但不应默认变成三套读者页面。

默认形态应是：

- 主阅读页是读者第一次阅读入口，承载导读、当前位置或阅读地图、主体正文和必要关联说明；旧 `重点与亮点` 不再默认输出，设计提炼只有在具体且有助理解时才进入读者页。
- 机制说明只在脚本、模板、hook、状态文件、外部依赖或运行约束确实影响理解时按需生成；没有真实机制时不造机制页。
- 证据附录不作为普通阅读入口；完整源文、源块、hash、行号、覆盖数据和机器执行视图默认留在 `manifest.json`、`source.md` 或审计层。
- Codex 必须吸收非正文但有设计价值的部分，包括 frontmatter、hooks、脚本、模板、测试、权限、工具限制、目录结构、输出格式、失败处理和验证方式，并把它们转成设计说明、机制边界和可迁移经验。

书/长文与 Skill 包要分开处理：书/长文的章节正文应完整保留，不能按 Skill 执行壳逻辑剥离；Skill 包可以剥离执行壳，但必须保留用途、触发场景、核心流程、判断标准、约束、失败条件、输出契约和设计亮点。

页面大改会改变人工阅读体验。任何已标记 `human_status=accepted` 的页面在主体结构重写后，不能自动继承人工验收；应降回 `pending` 或标记需要重新验收，除非人工明确确认新版仍通过。

## D-023 “重点与亮点”下线，旁通视角暂不进入默认生成器

当前“重点与亮点”机制把三件事混在一起：原文金句摘录、材料价值提炼、跨材料或跨行业的高阶启发。这个混合定义会导致输出退化为几句醒目原文和普通旁批，不能稳定服务复杂材料陪读。

后续应拆分为不同能力：

- 可划线句：忠于原文，只标出真正值得回看的句子，数量可以为 0，不强行凑数。
- 材料理解辅助：帮助读者进入正文结构、判断链条和阅读路线；这部分已经由导读、当前位置、知识地图或正文结构承担，不应重复摘要。
- 旁通视角：在读者需要时，引入其他行业案例、思维模型、既有读书经验、LifeAtlas 知识或外部搜索结果，提供新的理解角度。

旁通视角不应在 v0.1 默认导入阶段自动生成，也不应固定套用某几个名人或模型。它需要可靠来源、明确触发点和清晰读者收益。更合适的后续形态是独立 Skill、读后批注/提问触发能力，或基于 LifeAtlas/思维模型库/联网搜索的单独模块。

在该能力设计清楚前，不应继续扩展现有 `highlights` 字段来假装解决问题。当前生成器已从主阅读页、validate 和测试中移除 `重点与亮点` 默认功能；旧 JSON 里的 `highlights` 字段只作为历史产物保留，不作为生成依据。若以后需要恢复相关能力，应先定义样张、来源和触发机制，再决定 schema 和生成器改造。

## D-024 Skill 源先更新本地 canonical package，LifeAtlas 采用版本化试产

Skill 包是活资产，upstream 可能改变触发条件、工具边界、hooks、脚本和正文语义。ReaderLab 生产前必须做版本预检：记录本地版本、远端版本、commit/tag、dirty 状态、Skill 新增/删除/改名/正文变化和支持文件变化。

生产新版本时，先把本地 canonical package 更新到 upstream 最新版，再以本地新版作为 ReaderLab 的唯一输入。不要长期依赖 `/private/tmp` 快照作为正式生产源。

upstream 是默认主事实。旧本地包里的非 upstream Skill 不默认保留，也不因为旧 `PACKAGE.json` 写过 `preserve` 就自动进入新版。ReaderLab 不替上游开发者背历史包袱。

只有满足以下条件时，非 upstream Skill 才能作为本地扩展进入新版：

- 它是用户或本项目明确单独开发、仍想维护的扩展。
- upstream 最新版没有同名、等价、合并或替代能力。
- 用户明确批准把它作为“本地扩展”保留。
- 阅读包中必须标明它不是 upstream 官方 Skill。

LifeAtlas 阅读包仍采用版本化试产：

- 旧阅读包保留为旧版本事实和旧批注载体，不直接覆盖。
- 新版阅读包以独立 book id 生成，例如 `dbs-suite__v2.15.1_096f726`，或由用户确认后再提升为主版本。
- 新旧版本之间生成差异报告，说明新增、删除、变更、被移除的旧本地项、经用户批准保留的本地扩展，以及需要人工复核的页面。
- 新版人工验收通过后，再决定是否把新版提升为当前主版本；提升前不删除旧版。

冲突处理规则：

- Skill 未变化：可沿用旧阅读判断，但新版仍记录来源版本。
- Skill 正文或支持文件变化：新版重新生成，人工状态默认为 `pending`。
- hooks、allowed tools、脚本、模板或状态文件变化：关联说明和阅读地图必须重新生成，旧人工验收不继承。
- Skill 改名或移动：先用 manifest / 差异报告建立映射；批注能定位再迁移，不能定位就保留在旧版或进入待处理。
- Skill 删除或不再属于 upstream：旧页只留在旧版本，新版主路线不出现，除非用户批准它作为本地扩展保留。
- 新增 Skill：进入新版主控清单，默认人工待验收。

本地 canonical package 更新的最低要求：

- 更新前先做只读版本预检和差异报告。
- 更新命令以 upstream 最新版为准；旧本地非 upstream Skill 默认移出新版输入。
- 更新后再次检查本地版本、Skill 列表和差异，确认本地事实已经从旧版切到新版。
- ReaderLab 只读取更新后的本地 package，不在生产步骤中混用远端快照和旧本地目录。

## D-025 v0.1 基础闭环后，优先建设全局理解层和精华提炼层

2026-06-29 的 gstack、dbs-suite 和《埃隆之书》Demo 后，ReaderLab v0.1 的产品闭环可以视为基本跑通：复杂材料可以被清洗、中文化、切段/分组，组织成 Obsidian 可读的 Markdown 阅读包，并验证真实 `tandem-comments` 批注链路。

下一阶段不应继续把主要精力放在扩大“清洗、翻译、切段”的加工规模上。ReaderLab 的产品升级方向是从“可读材料生产”进入“全局理解与精华提炼”：

- v0.4 全局理解层：为图书生成全书地图，为 Skill 包生成能力地图。地图回答核心问题、主线结构、章节/Skill 功能关系、概念/方法/案例关系、推荐阅读路线和待复核判断。
- v0.5 精华提炼层：参考仓颉 `book2skill` / book-to-skill，从材料中提取框架、原则、案例、反例、术语、可迁移洞察和候选 Skill 化方向。
- v0.6 升格层：只有经过全局地图和精华提炼验证的内容，才进入方法卡、概念卡、工作流、Skill 草案或 LifeAtlas 正式沉淀候选。

边界：

- 全局地图不是摘要。摘要压缩内容；地图解释结构、关系、主线和可复用价值。
- 精华提炼不是自动生成正式 Skill。第一阶段只产出候选资产，必须保留来源、适用边界、置信度和待复核项。
- 只读 TOC、前言或局部章节时，地图必须标明来源范围和置信度，不能假装已经读完整本书。
- 批注插件增强、复杂 registry、HTML 展示、外部集成和自动升格都后移；当前已有 manifest、machine/human 状态、版本化 book id、批注保护和运行账本，足够支撑 v0.4/v0.5 的样张验证。
- 旁通视角、跨材料比较、外部搜索或 RAG 仍按需设计，不进入默认生成器页面。

## D-026 v0.4/v0.5 先做数据契约，不先堆生成器模板

技术侧独立评估后确认：当前 `scripts/readerlab.py` 已经是 v0.1 压力样本驱动的单体脚本，内部混有通用生成逻辑、样本特定路由、页面清洗、状态判断、validate 和批注处理。它可以支撑 v0.1 验证，但不适合作为 v0.4/v0.5 的继续堆叠位置。

下一阶段全局理解层和精华提炼层必须先定义外部数据契约，再决定生成器实现。最低契约包括：

- `global-map.v1`：材料范围、来源覆盖、核心问题、结构主线、章节/Skill 功能关系、概念/方法/案例关系、推荐阅读路线、置信度和待复核项。
- `distillation.v1`：框架、原则、案例、反例、术语、可迁移洞察、候选 Skill 化方向、适用边界、来源范围、置信度和人工状态。

实现原则：

- 先用《埃隆之书》和 `dbs-suite__v2.15.1_096f726` 做 JSON/Markdown 样张，验证读者价值和评审标准。
- Markdown 是展示层，不是事实层；事实层应保留来源范围、置信度、待复核项和人工状态。
- 不把 dbs、gstack 或 Elon Demo 的专用结构直接写成 ReaderLab 通用规则。
- `delivery_status=deliverable` 只表示机器包可交付，不表示人工阅读质量通过。后续如继续扩展状态，应考虑改名或新增更明确的人工验收检查。
- 页面清洗、正文保留和代码块处理应按材料类型分流：Skill 包、书籍、长文和代码文档不能共享同一套执行壳剥离规则。

## D-027 下一阶段采用调度型主 Agent，执行和测试交给子 Agent

下一阶段 ReaderLab 工作切换为“主 Agent 调度 / 子 Agent 执行”的协作模式。

主 Agent 的职责：

- 读取最小真相层，确认当前事实、范围、约束和决策背景。
- 把目标拆成边界清晰、可验收的子任务。
- 分派子 Agent / worker，明确每个子任务的读写边界、禁止事项和验收证据。
- 收回子 Agent 结果，审查 diff、产物、命令输出和风险说明。
- 向用户解释取舍、汇报状态、提出需要用户决策的问题，并做最终交付判断。

子 Agent / worker 的职责：

- 负责具体代码修改、文件生成、批量执行、validate、测试、LifeAtlas 写入和报告整理。
- 在自己的结果中列明改动文件、执行命令、验证结果、失败点和未解决风险。
- 不擅自扩大写入边界，不修改 gstack 或 DB Skill 源包，不写 LifeAtlas `300/600/800`，不新增依赖，除非主 Agent 已确认用户批准。

主 Agent 在该模式下不直接进行代码修改、文件生成或测试执行。验收需要新证据时，主 Agent 应派发子 Agent 补跑，而不是自己下场执行。

这个模式的目的不是降低主 Agent 责任，而是减少主上下文冗余和执行污染。子 Agent 完成不等于任务完成；最终范围判断、风险判断和交付判断仍由主 Agent 负责。

## D-028 ReaderLab 是 Skill / Agent 工作流，脚本只是工具层

ReaderLab 不应被定义为一个不断扩大的生成器脚本。它的产品形态应是“复杂材料陪读 Skill / Agent 工作流”：用 Agent 的理解、判断、取舍和解释能力粘合复杂阅读过程，用脚本处理重复、稳定、可检查的工具动作。

职责分层：

- Agent 智能层：判断材料类型，理解材料结构，制定阅读路线，生成全书地图/能力地图，做精华提炼，解释来源范围、置信度、待复核项和人工验收边界。
- 脚本工具层：扫描文件、抽取目录、生成或更新 manifest、保护批注、写入契约实例、执行 validate、做批量稳定检查。
- 数据契约层：定义 `global-map.v1`、`distillation.v1` 等事实层最低标准，保证地图和提炼不是漂亮摘要，而是带来源、边界和复核状态的可检查产物。
- 展示层：LifeAtlas / Obsidian Markdown 阅读包，服务人的第一次阅读、批注和讨论。

后续技术推进不能把“把所有逻辑塞进 `scripts/readerlab.py`”当作默认方向。正确路线是先沉淀 ReaderLab Skill 的主流程，再把其中确定、重复、低判断含量的步骤下沉为脚本工具。

边界：

- 全书地图、能力地图和精华提炼需要 Agent 判断，不能完全按模板机械生成。
- 脚本可以检查最低结构、来源范围、状态和批注保护，但不能证明阅读质量。
- 材料类型不同，Agent 流程和脚本工具调用顺序也可能不同；书、长文、代码文档、Skill 包和混合材料不能共享同一套机械清洗逻辑。
- 只有当某个步骤在多个材料上稳定重复、判断空间较低、失败模式清楚时，才适合沉入脚本。

## D-029 采用薄骨架换轨，不整体重构

ReaderLab 的方向需要从“生成器脚本项目”换轨为“复杂材料陪读 Skill / Agent 工作流”，但当前不做整体重构。

原因：

- v0.1 的清洗、切段、Markdown 包、批注保护和 validate 链路已经能工作；一次性大拆会破坏已验证基础。
- 第一轮《埃隆之书》全书地图和 `dbs-suite` 能力地图质量未通过，说明当前更缺的是阅读方法、契约和评估，不是更多代码。
- 在没有高质量样张证明前，提前拆脚本会把架构讨论变成无验证的工程运动。

当前路线：

- 先产出 ReaderLab Skill IR v0，定义能力边界、触发/不触发、工作流阶段、Agent/脚本职责和验收方式。
- 再产出 output eval v0，用《埃隆之书》和 `dbs-suite` 建立样张质量评估。
- 再补 `source-registry.v1` / `location-map.v1`、`catalog-map.v1` / `grounded-global-map.v1`、`capability-map.v1` 等薄契约。
- 最后按新流程重做样张；只有样张质量明显提升后，才逐步把稳定重复的部分从 `scripts/readerlab.py` 拆成工具模块。

旧样张处理：

- 《埃隆之书》旧 `02_全书地图.md` 不能作为合格全书地图。低覆盖情况下只能生成目录地图或局部地图；真正全书地图必须基于完整正文覆盖。
- `dbs-suite` 旧 `06_能力地图.md` 不能作为合格能力地图。Skill 包应使用专门的 `capability-map.v1`，而不是硬套图书 `global-map.v1`。

## D-030 用 yao-meta-skill 完善 ReaderLab Skill 工程化

`yao-meta-skill` 在 ReaderLab 中的作用不是读书、解析材料或生成阅读页，而是帮助把 ReaderLab 自身做成可复用、可评估、可治理的 Skill。

采用内容：

- Skill Engineering Method：先判断是否值得做成 Skill，再定义 job、输出、排除项、边界、参考扫描、风险和 gate。
- Skill IR Method：先写平台无关能力契约，再决定 Codex Skill、scripts、schemas、reports 的落地形态。
- Output Eval Method：用 baseline vs with-skill 的方式证明 ReaderLab 是否提升最终用户可读结果，而不是只证明路由或 validate 通过。

ReaderLab 的最小应用：

- `ReaderLab Skill IR v0`：记录 recurring job、触发条件、不触发条件、near-neighbor、工作流阶段、Agent/脚本边界、资源、脚本、报告、eval 计划、风险和成熟度。
- `ReaderLab output eval v0`：至少包含《埃隆之书》和 `dbs-suite` 两个 case，断言来源范围、覆盖率、不得假装全书读完、结构解释、精华 source refs、边界/反例、读者路线和人工待复核状态。
- `output-risk-profile`：提前列出样张常见失败，如泛泛摘要、目录冒充地图、低覆盖冒充全书、精华无来源、Skill 包能力地图退化成目录清单。

边界：

- 不把 `yao-meta-skill` 的治理、发布、registry、安装和多平台适配全量引入 ReaderLab。
- 不把 Skill 工程化等同于正式安装 ReaderLab Skill；当前先做 IR 和 eval。
- 不因引入 `yao-meta-skill` 而新增依赖或修改全局 Skill 安装状态。

## D-031 IR / eval / 契约薄骨架先于样张重做

ReaderLab Skill 化换轨的第一步不是改生成器，也不是继续修补旧样张，而是先形成可审查的 IR、output eval 和契约薄骨架。

当前采用三份设计产物作为下一阶段 worker 的依据：

- `docs/readerlab-skill-ir-v0.md`：回答 ReaderLab 负责什么、何时触发、何时不触发、哪些资源和脚本承载行为、哪些 eval 证明有效。
- `docs/readerlab-output-eval-v0.md`：用《埃隆之书》和 `dbs-suite` 建立 baseline vs with-ReaderLab 的质量断言，旧样张只作为负面 baseline。
- `docs/contracts/readerlab-contracts-v0.md`：先拆出 source/location、catalog/global、capability 三类薄契约，再决定是否实现 JSON Schema 和 validate。

这些产物的状态是 `draft-v0`。它们足以派发 worker 开始重做样张，但不代表 ReaderLab Skill 已正式安装、生成器已重构、provider-backed eval 已运行，或旧样张已通过人工阅读质量验收。

后续只有当新版《埃隆之书》和 `dbs-suite` 样张通过 output eval 断言并完成明确人工复核后，才讨论把稳定、低判断含量的检查沉入 `scripts/readerlab.py` 或拆为工具模块。

## D-033 参考项目必须转成可验收的 ReaderLab 能力改动

ReaderLab 已经调研并登记了多类 GitHub 项目、本地 Skill 包和方法资产，但“写进研究日志”不等于“被 ReaderLab 吸收”。后续所有参考吸收都必须以实际能力改动为验收对象。

本决策里的“参考项目”特指会影响 ReaderLab 产品能力、阅读输出、来源契约或 Skill 形态的仿型参考。Meta Skill、CTK、MEM 属于工程审查和项目协作方法，不作为阅读输出的仿型参考，也不应进入内容能力吸收清单。

采用规则：

- 每个参考项目或 prompt 资产必须写清：借什么、不借什么、翻译成 ReaderLab 的设计是什么、落到哪个具体产物、如何验证效果。
- 候选来源不预先封死：先复查已有研究日志、本地 canonical Skill、本地 GitHub 能力库和 prompt 资产；如果不足以解决 ReaderLab 原始目标，再扩展本地能力库查询或直接搜索 GitHub。
- 吸收矩阵必须记录被放弃的强相关候选及原因，避免因为当前 prompt 没点名就遗漏已调研资产。
- 可落地产物包括：ReaderLab Skill IR、工作流阶段、契约字段、输出页面结构、source/location refs 粒度、output eval 断言、review gate、worker 任务说明、脚本工具边界。
- 参考吸收必须能被样张反测。若《埃隆之书》或 `dbs-suite` 输出没有变得更深、更可追溯、更接近读者需求，不能把该参考视为已经吸收。
- 参考项目不得直接引入新依赖、安装或启用外部 Skill、改全局状态，除非用户另行批准。

当前优先级：

- 仓颉/拆书类参考优先解决：总收获过浅、缺案例/反例/术语/可迁移洞察、缺三重验证。
- source-grounded / NotebookLM 类参考优先解决：来源范围、位置引用、覆盖率、精华和亮点可追溯性。
- Agent Skills 类参考优先解决：Skill 触发边界、资源分层、输出评估、risk gate、产品 ready 与 demo gate 分层。
- Meta Skill、CTK、MEM 只作为执行和审查方法：Meta Skill 负责 Skill 级评估口径，CTK/MEM 负责调度、状态和收尾纪律；它们不进入 ReaderLab 阅读输出的仿型参考列表。

## D-034 短板驱动吸收，不做仿型功能拼装

ReaderLab 学习外部 Skill、GitHub 项目或 prompt 资产时，不再从“对方有什么功能”出发，也不再把参考项目当作可拼接功能菜单。

正确顺序是：

1. 先识别 ReaderLab 当前原型的具体短板。
2. 再找在该短板对应能力环节上表现最强的仿型。
3. 学它的解题逻辑、判断链和失败防线，而不是照搬字段、页面形态或所有加分项。
4. 用同一材料做小范围重考，证明 ReaderLab 在该短板上变好。
5. 验证有效的逻辑进入当前生产链；暂不相关的独特能力只登记为未来储备。

执行含义：

- 不能再用“我们参考了仓颉/乔木/book-to-skill”作为进度证明。
- 必须说清楚：ReaderLab 哪个模块弱，谁在这个模块强，学它什么逻辑，落到哪个产物，如何评分。
- 同一个仿型可能在 A、B 环节强，在 C 环节一般；不能把项目整体打成“采用”或“不采用”。
- 最终读者产品不能露出多个仿型模块拼接痕迹；积木化的是生产能力，不是页面外观。

## D-035 结构化提炼是一套核心能力，不拆成地图/章节/亮点三套

2026-06-30 方向重设后，本决策的有效范围收窄为“结构化提炼能力如何理解”。它不再定义 ReaderLab 的下一阶段主线；下一阶段主线以 D-040 到 D-044 为准。

全书地图、章节地图、重点/亮点提炼背后是同一套核心能力：结构化提炼。

结构化提炼解决的是：在一堆材料中识别主干、支撑、案例、噪音、高价值增量和边界，并把它组织成读者能理解的结构。

不同输出只是粒度不同：

- 全书级：主问题、结构递进、核心张力、关键模块。
- 章节/部件级：局部功能、与全书关系、内部推进。
- 点状提炼：最值得带走、迁移、警惕或降级的判断。

执行含义：

- 如果后续需要补结构化提炼，应把它作为 AI 陪读层中的局部能力处理，而不是让它支配产品形态。
- 仓颉优先作为筛选/升格/降级逻辑来源。
- 李继刚式深读优先作为缺口/增量/机制传导逻辑来源。
- source-grounded 思路优先作为提炼来源范围和位置精度约束。
- 乔木共读和 book-to-skill 分别主要服务片段深读/误读防护、能力化判断；不要混进结构化提炼的第一轮补洞，除非有明确收益。

## D-036 每轮大迭代必须更新模块化进度板

ReaderLab 后续不能再只报告“又完成一个样张”或“下一步马上产品 ready”。每轮大迭代后必须更新 `docs/progress.md`。

进度板必须包含：

- 当前总进度点，满分 100。
- 本轮新增点数、扣减点数和净变化。
- 模块状态：close、working、working-low、blocked、not_started。
- 本轮哪些判断被证实，哪些旧判断被推翻。
- 下一轮最多能增加多少点，以及加分条件。

约束：

- 分数是工程管理估算，不是产品质量证明。
- 可以扣分；发现旧乐观判断错误时必须扣分。
- validator 通过、样张存在、子 Agent PASS 不能自动加分，除非它关闭了某个模块的真实风险。

## D-037 Meta Skill 可作为横向能力评估裁判

Meta Skill 不作为 ReaderLab 阅读输出的仿型，也不替代 ReaderLab 自己的产品判断。但在横向比较 ReaderLab 与外部 Skills / GitHub 项目的能力时，可以作为评估裁判之一。

适用场景：

- 比较 ReaderLab 与多个仿型在某个生产环节上的能力差异。
- 判断参考扫描是否过宽、是否把功能菜单当成吸收。
- 评估 output risk、gate、promotion 条件和下一轮最高价值改进。
- 审查新生产链是否只奖励字段存在，而没有提升最终读者输出。

执行规则：

- 横向评估 worker 必须读取 `yao-meta-skill` 的相关方法文件，至少包括 `skill-engineering-method.md` 和 `output-eval-method.md`，必要时读取 `skill-ir-method.md`。
- 报告必须分清：立即采用、未来储备、不吸收。
- Meta Skill 报告只是一个裁判视角，主 Agent 仍需结合 ReaderLab 根目标、用户判断、样张证据和项目约束做最终取舍。
- 不因使用 Meta Skill 评估而安装、启用或修改任何 Skill，也不新增依赖。

## D-038 抽取仿型环节前必须做上下游依赖与组合增益审查

短板驱动吸收不等于可以把任何外部 Skill、方法或仿型拆成独立零件随意拼接。一个方法环节可能是为它的上一环输入、下一环判断、整体语境或隐藏质量门服务的；脱离原链路后，该环节本身可能失效，也可能让后续输出质量下降。

即使单个环节没有明显破坏上下游，也不能默认多个好环节放在一起就会形成好系统。ReaderLab 必须证明组合后的链路产生 **组合增益**：输出质量、判断闭环、读者收益或来源可靠性，至少有一项比单一仿型路线和旧 ReaderLab 输出更强；否则只能登记为并列参考，不能进入生产链。

因此，ReaderLab 后续学习某个仿型的局部环节前，必须先做 **方法依赖与组合增益审查**：

1. 说明拟抽取的环节在原方法中服务哪个上下游目标。
2. 判断它的输入是否依赖上一环的产物、判断标准或语境。
3. 判断它的输出是否必须进入下一环才成立，或单独使用会不会造成误判。
4. 判断 ReaderLab 是否已经有等价上下游承接；没有承接时不得直接吸收。
5. 说明多个环节拼接后新增了什么能力，而不是只把各自优点并排陈列。
6. 用小范围重考验证组合后输出质量优于旧 ReaderLab 和任一单一路线；不能只看字段是否齐全或页面是否更顺。

执行含义：

- “只学某一环”默认是待证假设，不是合理前提。
- 若环节强依赖原方法上下游，应改为整段吸收、延后吸收，或只登记为未来储备。
- 若两个仿型环节被拼接，必须说明它们之间的数据、判断和质量门如何衔接。
- 评估报告必须把“可独立抽取 / 需连同上下游抽取 / 不应抽取 / 组合无增益”说清楚。
- 若组合没有 1+1>2 的证据，不得把它说成生产链；最多说成多个参考视角。

## D-039 组合增益必须用同题多路线对照证明

ReaderLab 不能凭主观感觉判断“组合路线更好”。后续凡是要把多个仿型环节组合成生产链，必须做同题多路线对照。

最小对照形态：

1. 单一路线 worker：仓颉-only、李继刚-only、ReaderLab 当前链路；source-grounded 可作为独立约束路线或作为所有路线的共同来源门，视任务定义而定。
2. 组合路线 worker：至少一个“李继刚定性 + 仓颉筛选 + source-grounded 边界”组合版本；必要时增加二元组合版本。
3. 评审 worker：不参与生成，只按同一 rubric 比较输出质量。
4. 主 Agent：审查输入一致性、输出契约一致性、评审证据和最终取舍。

对照必须满足：

- 同一材料、同一材料范围、同一输出契约。
- 输出至少覆盖主问题、结构传导、候选池/选点、降级/拒绝、来源边界。
- 评审必须比较：目录复述减少、缺口说明、结构传导、选点理由、不升格理由、来源边界、读者可用性。
- 组合路线只有在明显优于单一路线和旧 ReaderLab 输出时，才可进入生产链。

如果组合路线没有胜出，ReaderLab 必须给出倾向性处理方案：选择表现最强的单一路线作为主方法，其他仿型只作为辅助 gate、储备或不吸收，不能为了“集齐优点”继续拼接。

## D-040 外部强模型触发方向重估时，当前任务降级为证据

`docs/current-task.md` 只记录当前执行切片，不是项目方向权威。当用户引入外部专家、强模型或新的产品判断，并明确要求重估方向时，旧 current task 必须降级为现状证据的一部分。

方向重估时的优先级是：

1. ReaderLab 根目标。
2. 用户当前明确判断。
3. 外部强模型建议及其可验证理由。
4. 仓库事实、旧样张、旧任务和运行记录。
5. 旧 `current-task.md` 的下一步切片。

执行含义：

- 不能因为旧任务写了“下一步”就默认继续执行。
- 旧任务如果只服务局部能力，而新判断指向产品终局或路径重设，应先重设方向。
- 方向重设完成前，不应继续推进旧局部任务。

## D-041 ReaderLab 采用正文优先陪读包作为产品形态

ReaderLab 的目标产物是正文优先陪读包：

- 一手正文轨：处理过但不被 AI 摘要替代的材料正文。
- AI 陪读轨：全局讲解、高阶讲解、深读卡、误读防护、批注讨论入口和候选沉淀。
- 设计资产轨：面向产品负责人的技术负责人解剖和可复用设计卡。

AI 不替代人的第一次阅读，也不能把提炼挡在正文前面。读者必须能接触足够完整的一手材料，再决定是否采纳 AI 的解释、升格或迁移建议。

执行含义：

- 主阅读页先服务材料阅读，不先展示 AI 结论墙。
- audit、hash、路径、命令、完整执行视图和机器状态默认进入事实层或附录。
- 候选沉淀只进入候选区，不自动写入 LifeAtlas `300/600/800`。

## D-042 AI 阅读方法采用 E-R-D-D

ReaderLab 的 AI 方法层采用 E-R-D-D：

```text
Evidence -> Route -> Deepen -> Decide
```

- Evidence：建立来源、覆盖、位置和不能下结论的边界。
- Route：判断材料类型、阅读单元和主体/支撑/外壳/实现/审计分层。
- Deepen：生成主问题、机制链、原则、案例、反例、术语、技术设计、失败模式、迁移洞察和误读风险等候选。
- Decide：通过 Source、Structure、Reader Gain、Transfer/Technical、Overreach gate，决定 Promote、Keep、Downgrade 或 Reject。

执行含义：

- 任何旧组合路线只能作为该链条某个局部实现，不是产品终局。
- AI 不能先写漂亮全局结论，再事后补引用。
- 覆盖不足时只能输出目录地图、局部地图或结构诊断。
- 真正升格的深读内容必须有来源、读者收益、边界和人工状态。

## D-043 技术材料必须有技术负责人 / 设计资产提炼层

Skill 包、代码文档、Agent 工作流、Prompt 系统和工程资料不能只做中文化、摘要或目录式能力地图。ReaderLab 必须提供技术负责人 / 设计资产提炼层，解释设计原子的作用、代价、风险和迁移边界，并把隐藏工程设计翻译成产品负责人可复用的设计资产。

设计原子包括但不限于：

- trigger
- user_intent
- inputs
- outputs
- workflow
- tools
- state
- constraints
- failure_modes
- human_gate
- output_contract
- verification
- risk_control
- transferable_pattern
- non_transferable_part

执行含义：

- 不允许只复述 Skill 文档。
- 不允许只夸设计精妙。
- 必须说明什么时候不要用，迁移时最可能坏在哪里。
- 必须区分原文显性说明和 AI 工程推断。
- 必须说明它对产品负责人有什么可复用价值，以及能沉淀成什么设计卡。

## D-044 正式开发采用 contract-first，不直接开发完整生成器

正式开发的第一步不是完整自动生成器，也不是继续扩写样张，而是 contract / validator / two-sample proof 的最小闭环。

下一阶段顺序：

```text
Phase 1：contract / validator
Phase 2：两个最小样本
Phase 3：Markdown renderer
Phase 4：output eval
Phase 5：Agent workflow
Phase 6：半自动生成器
Phase 7：人工确认后的沉淀与 Skill 化
```

最低 contract 包括：

- `source-registry.v1`
- `location-map.v1`
- `catalog-map.v1`
- `local-deepread.v1`
- `capability-map.v1`
- `output-eval.v1`

执行含义：

- Markdown 是展示层，不是事实层。
- validator 只证明机器规则通过，不能证明人工阅读质量通过。
- 两类最小样本必须覆盖书籍/长文和 Skill/工程材料。
- 只有多个材料类型上的稳定、低判断动作，才适合沉入脚本。

## D-045 读者页必须提供方位感，不暴露内部路线和审计结构

2026-06-30 的《埃隆之书》source-aligned 长样张证明了来源追溯可以成立，但人工反馈指出读者体验没有通过：读者页暴露了 `source refs`、`heading path`、`human_status`、降级/拒绝理由等内部审计语言；“阅读路线”“主题线索”“处理过的一手正文”等表达也不像真实读书介绍。

因此，ReaderLab 的读者可见产品原型必须从“内部分析报告”改成“材料方位感 + 正文陪读”：

- 包级入口先回答：这份材料讲什么、为什么值得读、它怎样组织、当前覆盖到哪里、读者先看哪一部分。
- 全书/材料地图回答：整体主问题、主要部件、部件之间的关系、当前章节/模块在整体中的位置。
- 章节/模块页回答：这一章讲什么、为什么出现在这里、承接前面什么、为后面什么服务、读者读完应把它放回哪个全局结构。
- “阅读路线”只保留为 Agent/contract 内部结构判断，不作为读者页标题或核心产品概念。
- “主题线索”改为更自然的读者语言，例如“这本书在讲什么”“全书结构”“这一章在全书中的位置”“读完本章要带走什么”。
- 书籍/长文读者页默认原样保留正文；除非用户明确要求整理原文，否则只做空格、空行、断行等轻量清理。AI 重述不能冒充“一手正文”，也不能用导读替代正文。
- `source refs`、claim trace、降级/拒绝清单、机器状态、覆盖审计和内部 gate 默认进入 `audit/`；读者页只保留必要的自然语言误读提醒。

执行含义：

- source alignment 通过只能说明审计层可追溯，不能说明读者原型通过。
- 下一阶段开发必须先与读者原型对齐，再继续第二类材料迁移测试、renderer 扩展或半自动生成器。
- eval gate 必须检查读者页是否像人会实际使用的阅读包，而不是只检查字段、来源和 validator 通过。

## D-046 书籍正文原样保留，Skill 正文净化，技术负责人层沉淀设计资产

2026-06-30 读者复核后确认：ReaderLab 不能把书籍或长文压缩成 AI 导读，也不能把 Skill 包只做摘要。当前产品边界进一步明确为：

- 书籍 / 长文：默认原样保留正文和章节顺序；除非用户明确要求整理原文，否则只允许做空格、空行、断行等不改变意义的轻量排版清理。
- Skill 包 / 工程材料：默认生成净化正文，剥离安装命令、重复模板、执行外壳和机器噪音，但保留用途、触发条件、用户意图、核心流程、约束、失败条件、输出要求和设计亮点。
- 被剥离的命令、脚本、路径、hash、执行协议和机器证据不删除，进入附录、技术解析或 audit。
- AI 高阶讲解应像读完整份材料的人在讲清楚材料，而不是把内容切成固定栏目。
- 技术负责人层面向产品负责人，不是工程术语解释；Codex 应主动识别用户不容易感知的用户意图设计、问题建模、输入治理、约束设计、输出契约、失败路径、验证机制、可迁移模式和不可迁移边界。
- 技术负责人层最终应沉淀为可复用设计资产卡：适用场景、解决问题、原材料依据、可复用做法、使用前提、失败风险、什么时候不要用。

执行含义：

- 没有原样正文的书籍/长文 reader 页不能通过 reader experience。
- 没有净化正文的 Skill/工程 reader 页不能通过 reader experience。
- 没有设计资产提炼的技术负责人层不能通过技术材料验收。
- 后续样张、renderer 和 contract 设计必须先满足正文硬边界，再谈全局讲解、深读、迁移和自动化。

## D-047 正文优先陪读包采用输出契约作为样张前置门

2026-06-30 用户确认输出契约方向。ReaderLab 后续不再先打磨旧《埃隆之书》reader 页，也不直接开发完整生成器；必须先用输出契约约束下一轮样张。

输出契约包括：

- 书籍 / 长文：原样正文和章节顺序是主体；默认不压缩、不改写、不重述；只允许空格、空行、断行等轻量排版清理；AI 讲解放在正文之后或旁边；没有原样正文的一手轨，reader experience 必须失败。
- Skill / 工程材料：净化正文是主体；剥离安装命令、重复模板、执行外壳、机器状态、路径、hash 和调试信息；保留用途、触发条件、用户意图、核心流程、约束、失败条件、输出要求和设计亮点；被剥离内容进入 audit、附录或技术负责人解析。
- 高阶讲解：像读完整份材料的人给读者讲清楚材料；用一段或几段完整讲解，不把内容切成固定栏目；类比和跨领域视角必须服务材料理解。
- 技术负责人 / 设计资产提炼：面向产品负责人，主动识别用户意图设计、问题建模、输入治理、约束设计、输出契约、失败路径、验证机制、可迁移模式和不可迁移边界；输出可复用设计资产卡。
- Audit 分离：source refs、location map、claim trace、machine/human 状态、降级/拒绝记录进入 audit；reader-facing 页不把内部审计结构当主内容。

执行含义：

- 下一步是按契约选择一个小样本重做 reader-facing 样张。
- 旧《埃隆之书》第二部分 source alignment 可以保留为审计证据，但 reader experience 仍为 FAIL。
- 若样张违反输出契约，即使 source alignment 或机器验证通过，也不能进入 product ready 或 human accepted 口径。

## D-048 高阶讲解必须提供非摘要型认知增量

2026-06-30 用户进一步明确：ReaderLab 的 AI 高阶讲解不能停在“小总结”“本章重点”或普通书籍总结；但它也不能丢掉重点和要点。AI 辅助阅读的核心价值在于它拥有更大的知识背景和更强的联想 / 推理能力，应该像一个拥有全局视角、见识丰富的人在陪读。

用户列举的查理芒格式跨学科思维模型、历史观、旁征博引、产品 / 组织 / 工程经验和既往经历连接，表达的是一种需求方向，不是封闭实现清单。若后续能找到更完善的讲解结构，应优先服务“讲清重点 + 提供更大价值观和体系”的目标，而不是机械套用这些例子。

高阶讲解应当：

- 高屋建瓴地重看材料，指出局部正文背后的更大问题结构。
- 旁征博引，适度引入跨学科思维模型、历史观、商业案例、组织管理、工程系统、心理机制和用户既往经历中的相似结构。
- 解释看似不同领域之间的底层同构关系，而不是只做表面类比。
- 帮助读者判断哪些观点值得吸收，哪些只能降级，哪些必须拒绝。
- 明确区分原文直接支持、AI 解释性重组、外部类比和待验证判断。

同时，高阶讲解不得：

- 用导读、旁批、误读提醒等栏目墙替代自然讲解。
- 脱离正文堆名人、堆模型、堆历史例子。
- 把宏大世界观写成不受来源约束的 AI 结论。
- 只复述“本章讲了什么”就声称深读通过。

执行含义：

- real-source demo v1 只能说明正文优先契约方向基本成立，不能说明 ReaderLab 体验通过。
- 下一步应先重做 AI 高阶讲解口径和小样章，而不是继续加样本或开发生成器。
- eval gate 必须检查“非摘要型认知增量”和“跨领域连接是否反向照亮正文”。
