# 复杂材料阅读化 SOP

## 目标

ReaderLab 的目标不是把源文件机械搬进 Markdown，而是把复杂材料转成可以阅读、批注、讨论和沉淀的 LifeAtlas 阅读包。

本 SOP 适用于电子书、长文、Markdown 文档、课程资料、访谈稿、代码文档、Skill 包和混合型资料。gstack 和 dbskill 都只是压力样本，不是通用规则来源。

## 输入与输出

输入可以是一份材料、一个目录、一个 Skill 包或混合资料包。

输出默认分为读者层、Codex 吸收层和审计层。它们是逻辑分层，不等于默认生成三棵镜像目录：

- 主阅读页：给读者第一次阅读用，承载理解主线、阅读地图、正文主体和必要关联说明；旧 `重点与亮点` 不再作为默认段落。
- Codex 吸收层：给 Codex 理解非正文机制、回答批注、追溯来源和提炼可迁移经验。
- 审计证据层：保留完整原文、完整译文、代码、hash、来源路径和覆盖校验，默认不作为普通阅读入口。
- 批注线程：使用 LifeAtlas 当前 Obsidian 插件 `tandem-comments` 的真实格式保存读者批注和 Codex 回复。
- 旁通视角：后续可选能力，用于按需引入跨材料、跨行业、思维模型或导师式视角；当前不属于 v0.1 默认生成内容。

## 阅读路线规划

ReaderLab 先读整体结构，再规划阅读路线。文件名、目录名和关键词只能作为辅助证据，不能替代对入口、正文职责和材料结构的理解。

### Skill 包

Skill 包先按这个顺序判断：

1. 读包级说明、根 `SKILL.md`、总控 Skill 或路由 Skill，确认整包入口和推荐起点。
2. 读每个 `SKILL.md` 的 frontmatter、description、触发条件、正文职责和边界，判断它真正解决什么阅读问题。
3. 读同目录下脚本、模板、配置、迁移文件和说明文档的职责，判断它们是主体内容、必要关联说明、Codex 吸收材料还是审计证据。
4. 只有在入口、正文和支持文件仍不足以判断时，才把文件名和关键词作为兜底线索。

结构清楚时，生成阅读路线，并说明为什么这样读。结构复杂但能解释时，也生成路线，同时标出争议点和把握程度。

如果读完入口、正文和支持文件后仍不能形成稳定路线，输出“材料结构诊断”：说明入口缺失、职责混杂、路由矛盾或阅读价值存疑。不要把它塞进看似整齐但错误的目录，也不要把“请用户人工分组”作为正常出口。

### 图书、长文和课程

图书和长文不套 Skill 包逻辑。后续实现图书导入时，先读书名、副标题、作者说明、前言、目录、章节标题和章节关系，再决定阅读路线。

图书和长文的章节正文应完整保留在主阅读页。短导读、全书位置和章节关系用于帮助理解，不能替代原章节正文。出版信息、目录、注释、参考文献、术语表等非正文材料由 Codex 吸收，用于解释章节位置、论证路线、术语体系和可迁移读法；如果某个附录本身有阅读价值，应作为章节/附录阅读页保留，而不是当作执行壳剥离。

如果目录和章节能形成主线，路线应跟随书本自己的结构；如果目录混乱、章节职责混杂或主题跳跃严重，先输出材料结构诊断，说明是否值得继续处理。

## 标准流程

### 1. 先判断材料类型

先用内容功能判断，而不是用文件名、目录名或某个样本包字段判断。

常见类型：

| 类型 | 判断信号 | 主阅读页重点 |
|---|---|---|
| 外壳膨胀型 | 主任务很短，但周围有大量通用启动、配置、遥测、路由、权限或环境检查 | 抽出真正任务主线，把通用外壳移到机制说明或附录 |
| 执行流程型 | 有步骤、命令、产物、退出码、失败处理和验收 | 保留操作闭环：做什么、怎么跑、产物是什么、如何验收、失败怎么定位 |
| 判断框架型 | 主要内容是原则、审查框架、决策标准、问题清单 | 保留高价值框架，不压缩成短摘要 |
| 短护栏型 | 材料短，主要讲一个保护、规则或组合入口 | 保持短，讲清保护什么、谁负责、怎么启停 |
| 状态/报告型 | 材料定义存档、恢复、汇总、报告、索引或跨会话状态 | 保留数据结构、读写边界、可信来源和失败条件 |

### 2. 做五类内容分层

每个模块先做分层，再写阅读页。

| 内容类别 | 处理方式 | 判断标准 |
|---|---|---|
| 一手主体内容 | 必须进入主阅读页 | 思想、论点、流程、判断标准、约束、失败条件、输出要求 |
| AI 导读和阅读辅助 | 必须保留，但不能替代正文 | 降低阅读门槛，提示读法、阅读地图、阅读路线和结构关系；旧 `重点与亮点` 不再默认生成 |
| 关联说明 | 进入主阅读页或机制说明 | 外来名词、前置背景、相邻材料、脚本职责、模板关系 |
| 运行/机器细节 | Codex 吸收；仅在影响理解时进入关联说明或按需机制页 | 环境检查、权限、同步、遥测、路由、缓存、升级、会话状态 |
| 完整追溯材料 | 默认进入 manifest/source，不打断主线 | 完整原文、完整译文、代码、hash、来源路径、覆盖校验 |

## 主阅读页写法

主阅读页默认包含四类内容：

1. 导读：说明这个模块解决什么阅读问题，读者应该带着什么问题读。
2. 阅读地图：说明本页在整本书、整包材料或当前阅读单元中的位置、上游/下游关系和建议阅读路线。
3. 正文：忠实呈现主体内容，可以按中文阅读顺序重排，但不能丢失关键论点、流程、判断标准、约束、失败条件和输出要求。书/长文章节正文必须完整保留；Skill 包正文应剥离通用执行壳，但保留主体设计。
4. 关联说明：解释图表、脚本、模板、术语、外部背景、机制关系或材料间连接。只放理解正文必需的信息，不放源块、hash、行号、覆盖比例和后台验收数据。
可划线句、旁批或外部启发不进入当前默认生成器，除非后续先定义清楚来源、触发和验收标准。

“旁通视角”不是主阅读页默认段落。它的目标是给读者带来外部新知或更高维解释，例如其他行业案例、思维模型、既有读书经验或 LifeAtlas 中相关材料。它应按需触发：可以来自读者批注后的 Codex 回复，也可以作为后续独立 Skill。没有可靠来源时不生成，避免平庸解读或伪连接。

主阅读页不追求完整执行视图。完整不等于全塞进正文，清晰也不等于摘要化删掉关键信息。

主阅读页不应默认出现 `页面分层`、raw frontmatter、自动生成注释、重复 preamble、telemetry、session、routing、source block、行号、hash、源文/译文比例或后台验收 HTML 注释。

## Codex 吸收写法

Codex 吸收层回答这些问题：

- 这份材料里哪些非正文部分有设计价值？
- 它们说明了什么触发边界、工具限制、权限约束、失败处理、验证方式或输出契约？
- 哪些机制可以迁移到其他开发项目、Agent 工作流或产品设计中？
- 哪些只是执行壳、审计证据或机器噪音，不应打断读者阅读？

Skill 包中需要吸收的非正文材料包括但不限于：frontmatter、hooks、allowed tools、脚本、模板、测试、配置、迁移、目录结构、输出格式、失败处理和验证方式。

图书和长文中需要吸收的非正文材料包括但不限于：书名、副标题、前言、目录、章节结构、注释体系、参考文献、术语表、索引和出版信息。

吸收结果应带来源引用，进入 manifest 或内部记录，供 Codex 回复批注、解释设计和追溯来源时使用。

## 机制说明写法

机制说明回答这些问题：

- 哪些脚本、模板、配置或外部文件影响理解？
- 它们各自承担什么职责？
- 它们和主材料是什么关系？
- 哪些是当前模块必要机制，哪些只是包级通用外壳？
- 如果执行失败，读者应该从哪个机制节点开始查？

机制说明按需生成，不是每个模块默认生成。没有真实脚本、模板、hook、状态文件、外部依赖或运行约束时，不应单独造机制页。机制说明不贴完整源码，除非该源码本身就是阅读对象。

## 证据附录写法

证据附录负责防止信息流失和支持审计。它默认不作为普通阅读入口，可以包含：

- 完整原文。
- 完整译文。
- 完整代码块。
- block manifest、hash、来源路径。
- 生成器覆盖数据。
- 机器执行视图。

附录不承担第一次阅读入口。

## 批注与 Codex 回复

v0.1 的真实插件目标是 Obsidian `tandem-comments`。

该插件把批注线程保存为文件末尾的 fenced block：

````markdown
```tandem-comments
{
  "<id>": {
    "anchor": {
      "exact": "原文片段",
      "pos": 123,
      "prefix": "前文上下文",
      "suffix": "后文上下文"
    },
    "status": "open",
    "thread": [
      {"author": "Reader", "ts": "...", "text": "读者批注"},
      {"author": "Codex", "ts": "...", "text": "Codex 回复"}
    ]
  }
}
```
````

ReaderLab 的批注闭环要求：

- 读取 comment block。
- 用 `exact`、`prefix`、`suffix` 和 `pos` 定位正文附近上下文。
- Codex 回复只回答对应批注，不重写全文，不做文末长篇总结。
- 回复追加到同一个 thread。
- 如果未来更换 Obsidian 插件，必须重新读取插件真实存储格式，不能复用 `tandem-comments` 假设。

Markdown 原生 `==高亮==`、`%%评论%%` 和 callout 只能作为 fallback 探索，不能替代插件格式验收。

## 验收标准

一个阅读模块达到预览验收，需要同时满足：

- 主阅读页能让中文读者理解主体，而不是被外壳和代码淹没。
- 高价值主体内容没有被压缩失真。
- 运行细节没有丢失，只是转移到机制说明或附录。
- 导读、阅读地图和必要关联说明辅助阅读，但不替代正文。
- 旁通视角如被生成，必须有明确触发点、可靠来源或清楚的类比边界，不能固定套用名人或模型。
- 批注线程能由 `tandem-comments` 定位到正文并追加 Codex 回复。
- validator 通过只能证明结构和覆盖没有明显错误，不能替代人工阅读质量判断。

## 禁止项

- 不把 gstack 的 `{{PREAMBLE}}`、`preamble-tier`、标题结构写成通用规则。
- 不把 Skill 包的完整翻译要求误当成所有主阅读页的形态。
- 不把“变清晰”做成摘要化失真。
- 不用 Markdown 手搓批注格式替代 Obsidian 插件真实格式。
- 不在未确认前写入 LifeAtlas `300/600/800`。
- 不新增依赖来绕过现有 Markdown + Obsidian + Codex 形态。
