# ReaderLab Product Spec

## 一句话定义

ReaderLab 是复杂材料吸收系统：把书籍、长文、课程资料、代码文档、Skill 包和混合型资料转成 LifeAtlas 陪读包，让读者先读到可信的一手正文，再获得全局讲解、章节讲解、技术负责人解剖和可沉淀的设计资产。

## 不是什么

- 不是摘要器：不能用 AI 重点替代一手正文。
- 不是翻译器：中文化只是降低阅读门槛的一部分。
- 不是完整图书生成器：正式开发前不追求全自动整书生产。
- 不是知识库自动升格器：候选沉淀必须人工确认后才能进入 LifeAtlas 正式区。
- 不是 UI / 数据库 / 知识图谱项目：v0.x 继续使用 Obsidian + Markdown + Codex + 现有批注插件。

## 产品原则

1. 一手材料优先。AI 陪读不能挡在正文前面。
2. 书籍和长文默认原样保留正文，不压缩、不改写、不用 AI 复述冒充正文；除非用户明确要求整理原文，否则只允许做空格、空行、断行等轻量排版清理。
3. AI 负责降低阅读阻力，不替代人的第一次判断。
4. 高层判断必须有来源范围和位置引用。
5. 低覆盖只能输出目录地图、局部地图或结构诊断，不能冒充完整理解。
6. 技术材料必须解释设计取舍、代价、边界和迁移条件，并沉淀为产品负责人能复用的设计资产。
7. Markdown 是展示层，JSON contracts / audit / eval 是事实层。
8. machine validation 不等于 human acceptance。
9. 读者页必须像真实阅读产品，不暴露内部 source id、claim trace、机器状态或审计字段。

## 输出契约

ReaderLab 的输出先按材料类型确定主体，再决定 AI 陪读和审计放在哪里。

### 书籍 / 长文

- 原样正文和原章节顺序是主体。
- 默认不压缩、不改写、不重述。
- 只允许空格、空行、断行等不改变内容意义的轻量排版清理。
- AI 讲解放在正文之后或旁边，不能抢正文位置。
- 没有原样正文的一手轨，reader experience 必须失败。

### Skill / 工程材料

- 净化正文是主体，不是摘要。
- 剥离安装命令、重复模板、执行外壳、机器状态、路径、hash 和调试信息。
- 保留用途、触发条件、用户意图、核心流程、约束、失败条件、输出要求和设计亮点。
- 被剥离内容不删除，进入 audit、附录或技术负责人解析。

### 高阶讲解

- 不是小总结，也不是只做“本章重点”复述；它仍要讲清重点和要点，但目标是进一步提供读者靠自己短时间难以获得的认知增量。
- 像一个读过完整材料、见识足够丰富、能站在更大世界图景里思考的人，给读者讲清楚材料为什么重要、哪里危险、如何迁移。
- 用一段或几段完整讲解，而不是把内容切成导读、旁批、误读提醒等固定栏目。
- 可以旁征博引，引入跨学科思维模型、历史观、产品经验、组织管理、工程系统、商业案例和用户既往经历中的相似结构，但这些只是示例，不是封闭清单。只要能更好地照亮材料、建立更大的价值观和理解体系，就应使用更合适的知识结构。
- 跨学科视角必须把底层逻辑串起来：它要解释“为什么这里和那个看似不同的领域是同一个问题”，不能只是堆名人、堆模型或写漂亮类比。
- 高维讲解仍受正文约束：必须区分原文直接支持、AI 的解释重组、外部类比和待验证判断。

### 技术负责人 / 设计资产提炼

- 面向产品负责人，不是工程术语课。
- 主动识别用户不容易感知的用户意图设计、问题建模、输入治理、约束设计、输出契约、失败路径、验证机制、可迁移模式和不可迁移边界。
- 输出可复用设计资产卡，至少说明适用场景、解决问题、原材料依据、可复用做法、使用前提、失败风险和什么时候不要用。

### Audit 分离

- source refs、location map、claim trace、machine/human 状态、降级/拒绝记录进入 audit。
- reader-facing 页不把内部审计结构当主内容。

## 产品能力层

### 1. 来源与可信度层

回答：这些内容从哪里来，覆盖到什么范围，AI 判断有没有依据。

最低产物：

- `source-registry.v1`
- `location-map.v1`
- coverage status
- source refs
- machine_status / human_status

### 2. 一手正文层

回答：读者真正要读的材料是什么。

书籍和长文默认保留原文正文和章节顺序。ReaderLab 不主动压缩、改写或重述正文；除非用户明确要求整理原文，否则只做不改变内容意义的格式清理，例如删除多余空格、修复空行、整理断行。AI 概括、讲解和旁批只能进入 AI 陪读层，不能冒充一手正文。

Skill 包、代码文档和工程材料的一手正文是“净化后的正文”：剥离安装命令、执行外壳、重复模板、机器协议和噪音，但保留真正描述用途、触发条件、用户意图、核心流程、约束、失败条件、输出要求和设计亮点的文本。被剥离的内容不删除，进入技术负责人解剖、附录或 audit。

### 3. 全局讲解层

回答：这份材料整体讲什么，当前章节/模块在整体中的位置是什么，读者如何保持方位感。

低覆盖时只能生成目录/结构诊断和局部方位说明；完整覆盖且有来源引用时才允许生成 `grounded-global-map`。

读者可见表达应像一个已经读完材料的人在“讲一遍”：这本书 / 这个 Skill 包到底在讲什么，前后部分如何承接，当前章节为什么出现在这里。它不是标签表，也不是内部路线图。`Route`、`source refs`、`human_status`、claim trace 等内部词汇只进入 contract / audit。

### 4. 高阶讲解与深读层

回答：哪些内容真正值得停下来理解，为什么值得。

候选类型包括主问题、机制链、原则、案例、反例、术语、迁移洞察、误读风险和降级候选。读者页上的表达应优先用成段讲解把事情讲透，而不是把同一段材料切成“导读 / 旁批 / 防误读 / 问题”的固定栏目。

高阶讲解的质量标准不是“有没有总结到位”，而是读者是否获得了新的看法：能否从局部材料看见更大的历史、商业、组织、工程、心理或系统结构；能否把看似不同学科和经历中的同构问题串起来；能否判断哪些观点值得吸收、哪些只能降级、哪些必须拒绝。真正升格为长期资产的内容再进入 deepread card。

### 5. 技术负责人 / 设计资产提炼层

回答：技术/Skill/Agent 工作流为什么这样设计，有什么代价和边界，能否迁移。

该层面向产品负责人，而不是工程师授课。ReaderLab 应替用户看出其不容易感知的工程和系统设计：用户意图设计、问题建模、输入治理、流程拆解、约束设计、输出契约、失败路径、验证机制、可迁移模式和不可迁移边界。最终沉淀应是可复用的设计资产卡，不是技术术语解释。

### 6. 批注与讨论层

回答：读者如何在正文附近留下问题，Codex 如何读取附近上下文并回复。

当前优先适配 `tandem-comments`，不自建批注系统。

### 7. 候选沉淀层

回答：哪些内容可作为 LifeAtlas 卡片、方法、工作流或 Skill 候选。

当前只输出候选，不自动 promotion。

## 阶段路线

```text
Phase 0：产品规格冻结
Phase 1：contract / validator
Phase 2：两个最小样本
Phase 3：Markdown renderer
Phase 4：output eval
Phase 5：Agent workflow
Phase 6：半自动生成器
Phase 7：人工确认后的沉淀与 Skill 化
```

当前 Phase 0 已完成第一版，并在 2026-06-30 进一步确认正文优先输出契约。real-source demo v1 已证明正文契约方向基本成立，但用户判定《埃隆之书》高阶讲解仍像普通总结，不足以证明 ReaderLab 体验。下一步不是继续返工旧《埃隆之书》reader 页，也不是继续加样本，而是先重做 AI 高阶讲解口径，再用小样章验证。
