# ReaderLab 对抗性审查清单

> Authority boundary: 本文件是风险清单和审查材料，不是当前任务源。当前执行事实、下一步和停止条件以 `docs/current-task.md` 为准。

## 审查目的

这份清单整理 ReaderLab 在处理复杂材料时反复暴露的核心流程风险。它不是网络安全审查，也不是公开服务风险审查。

当前根目标仍是：把复杂材料转成 Obsidian 中可读、可批注、可让 Codex 围绕批注回复的 LifeAtlas 阅读包。

## 从已有文档归纳出的核心问题

### 1. 结构通过不等于阅读质量通过

来源：

- `docs/project-retrospective.md` 记录过 `benchmark-models` 首轮 block/hash 覆盖完整，但正文有模板残留和英文原句残留。
- `docs/decisions.md` 的 D-010、D-013 只证明机器覆盖和质量门能拦一部分坏产物，不证明阅读体验已经通过。
- `README.md` 和 `AGENTS.md` 都明确 `manifest.json` / validator 不能替代人工阅读质量判断。

问题：

validator 能检查结构、缺块、hash、英文残留和部分格式问题，但不能判断页面是否真的适合人读。

解决思路：

- 保留机器校验，但把它定义为底线，不定义为阅读通过。
- 后续状态模型应区分 `machine_completed` 和 `human_accepted`。
- 人工验收重点看：主体信息是否完整、是否被摘要化失真、阅读主线是否被执行外壳淹没、批注点是否自然。

当前状态：

- 已拆状态模型：manifest 的 Skill 记录保留旧 `status`，并新增 `machine_status`、`human_status` 和可选 `human_review`。
- `validate` 继续只代表机器验收；`human_status` 默认 `pending`，已知人工认可的 DB 三个核心页通过小型 registry 标记为 `accepted`。
- 当前报告仍只能说“机器 validate 通过”“人工待验收”或“人工已验收”，不能把机器绿灯说成阅读质量最终通过。
- 页面结构重写后，已人工认可页面需要重新确认；人工状态不能作为生成器可自动继承的普通派生字段。

### 2. 长材料容易被摘要化、漏译或贴回原文

来源：

- `docs/project-retrospective.md` 记录旧 `readerlab.skill-reading.v1` 曾把 `benchmark-models` 压缩成摘要式解读。
- D-009、D-010 要求完整译文、分块、source hash 和覆盖清单，目的就是防止长文被模型自动压缩。
- D-015 又进一步收窄：完整覆盖是证据约束，不等于主阅读页必须机械展示全部执行细节。

问题：

长材料有两类相反失败：

- 为了读起来短，把一手主体内容压缩成摘要，导致精髓丢失。
- 为了覆盖完整，把运行外壳、代码、命令和重复协议全塞进主阅读线，导致阅读主线被淹没。

解决思路：

- 证据附录要完整，主阅读页要忠实保留主体信息。
- 主阅读页可以重排、分段、加导读和旁批，但不能删判断节点、流程节点、约束、失败条件和输出要求。
- 代码、命令、脚本和模板应说明职责；完整内容放机制说明或证据附录。

当前状态：

- gstack 和 DB Skill 的样本已经证明“执行视图不等于阅读视图”的方向成立。
- 生成器已经把正式 LifeAtlas 200 重刷成 `10_中文精读` / `11_机制说明` / `12_证据附录` 三层页面结构，并通过 validate。
- 但用户在 Obsidian 中查看后确认：默认三棵镜像目录增加阅读负担，证据和机制不应默认压到读者面前。
- 下一版方向已校正为“主阅读优先、机制按需、证据退后”：三层是内部逻辑，不是默认读者目录。

新增风险：

- 如果只删除机制页/证据页，而不让 Codex 读取 manifest/source 中的非正文机制，批注回复会丢上下文。
- 如果按形式剥离所有命令和代码，会误伤 `make-pdf` 这类执行流程型材料的主体。
- 如果页面大改后继续沿用 `human_status=accepted`，会把旧版人工认可误套到新版页面。

### 3. 样本结构会污染通用规则

来源：

- D-014 明确 ReaderLab 面向复杂材料陪读，不是 Skill/gstack 专项。
- D-016 明确不能把 `{{PREAMBLE}}`、`preamble-tier`、gstack 标题结构写成通用规则。
- `docs/agent-run-ledger.md` 记录过 DB Skill 首轮被放进 `08_补充Skills索引`，说明旧分组仍带有 gstack 时代残留。

问题：

如果用某个样本包的目录、标题、字段或命名前缀做通用规则，换材料就会错。DB Skill 已经暴露过：核心材料被归成“补充索引”，validator 仍可能绿灯。

解决思路：

- 先读整体结构，再按内容功能决定阅读路线。
- 当前“通用阅读角色分类”只能作为 v0.1 过渡规则分类器，不是所有材料都必须套进去的固定书架。
- `reading_units` 规划应记录用途、判断理由、判断依据、把握程度和结构状态。
- 不适配时输出 `90_材料结构诊断`，不要硬塞进已有组，也不要把“请用户人工分组”当成正常出口。

当前状态：

- gstack 固定组已经移除通用地位。
- DB Skill 已重刷到 6 个阅读角色组。
- 材料包级路线规划产物已实现；后续风险是图书/长文导入时不能套用 Skill 包逻辑。

### 4. Obsidian 批注是核心数据，重建阅读包可能破坏闭环

来源：

- D-017 明确 v0.1 不自建批注系统，优先适配 Obsidian 插件真实存储格式。
- `docs/current-task.md` 和 `docs/dev-state.md` 已确认当前插件目标是 `tandem-comments`。
- 当前 `import-skills --force` 会重建目标材料包目录，可能覆盖文件末尾的 `tandem-comments` block。

问题：

ReaderLab 的闭环是“人读、人标、Codex 回复”。如果重建阅读包覆盖批注，等于破坏核心产品价值。

解决思路：

- 写入前扫描目标阅读包是否存在 `tandem-comments`。
- 没有批注时允许覆盖。
- 有批注时先做风险审核：保存批注 block、记录来源文件和 anchor，再决定迁移、跳过或请用户确认。
- 最低要求是批注数据不能丢：需要知道哪段文字被批注、对应哪个材料/章节、thread 内容是什么。

当前状态：

- 已在 gstack 和 DB Skill 路径实测 `tandem-comments` 读取和回复。
- 已实现重建前批注保护：默认拒绝覆盖已有 `tandem-comments`，显式 `--preserve-comments` 时迁移或保存待处理批注。
- 剩余风险：当前只覆盖 `tandem-comments`，不覆盖 Markdown 原生批注格式。

### 5. 源材料里的指令不能污染当前 Codex 行为

来源：

- ReaderLab 当前处理的 Skill 包里有大量“你必须”“不要”“下一步执行”。
- D-014 要求 ReaderLab 先理解材料，再把它转为阅读包；源材料不是当前会话的执行指令。

问题：

如果 Codex 在精读或回复时把源材料中的 agent 指令当成当前任务指令，就会从“阅读材料”滑回“执行材料”。

解决思路：

- 精读工作流中明确源材料是被引用材料。
- 页面表达区分“原文要求”和“阅读解释”。
- 后续可在 manifest 中标记 `source_contains_agent_instructions=true`，用于提醒精读和批注回复阶段隔离指令。

当前状态：

- 已作为流程风险记录，尚未进入生成器字段。

### 6. 厚页不是错误，过早拆分和过早摘要才是错误

来源：

- DB Skill 的 `dbs-xhs-title` 是 75 个公式的长库型材料。
- 用户已明确：如果一个 Skill 本身塞了很多主体信息，第一版 demo 可以先读；必要时再对单页做人工二次拆解，不应过拟合极端个例。

问题：

长库型材料不能简单删公式，也不一定必须在 v0.1 自动拆成很多页。真正要警惕的是两件事：

- 为了减薄页面，把公式库摘要化，丢失主体资产。
- 为了追求统一结构，在第一版就为极端个例引入复杂拆分机制，偏离主线。

解决思路：

- v0.1 demo 接受厚页，只要主体信息保留、导读能说明如何读、批注能定位。
- 如果人工阅读确认页面太厚，再做单页二次拆解或公式库附录。
- 不把 `dbs-xhs-title` 这种极端长库型个例当成立刻改生成器架构的理由。

当前状态：

- `dbs-xhs-title` 已生成 completed 预览页。
- 需要用户人工判断厚度是否可接受。

### 7. 临时目录验证不能替代 Obsidian 真实路径验证

来源：

- `docs/dbskill-readerlab-test-report.md` 记录过 DB Skill 早期只导入 `/private/tmp`。
- 用户已纠偏：v0.1 要在 Obsidian / LifeAtlas 200 路径中验证，`/private/tmp` 只能做机器试跑。

问题：

临时目录能证明脚本能跑，但不能证明用户能在 Obsidian 里读、标、让 Codex 回。

解决思路：

- 机器试跑可以在 `/private/tmp`。
- 产品验收必须落到 `/Users/tianqiang/LifeAtlas/200_原始资料/270_电子书与书籍资料/<材料名>`。
- 不写入 LifeAtlas `300/600/800`，但可以写入 `200_原始资料/270_电子书与书籍资料`。

当前状态：

- DB Skill 已落到 LifeAtlas 200。
- 后续预览和批注闭环默认以 LifeAtlas 200 为准。

### 8. 当前实现仍偏 Skill 包，非 Skill 材料尚未验证

来源：

- AGENTS、README、D-014 都写明复杂材料包括电子书、长文、Markdown 文档、课程资料、访谈稿、代码文档、Skill 包和混合型资料。
- 当前代码入口仍主要是 `import-skills`。

问题：

如果只在 Skill 包上推进，ReaderLab 会继续被 Skill 结构牵引，最终变成材料转换器或 Skill 专项工具。

解决思路：

- 这是上线前必须验证的方向，但不是下一步最高优先级。
- 下一阶段先把阅读单元规划补齐，再选一个非 Skill 材料做最小压力样本。

当前状态：

- 记录为后续风险，不作为当前下一步实现项。

## 当前优先级

1. 结构化输出：把主阅读页、机制说明、证据附录从 SOP 落到生成器结构。
2. 如需让正式 LifeAtlas 200 包显示新路线字段和清单文案，先由用户确认重刷范围，并继续保护 `tandem-comments`。
3. 再扩非 Skill 材料样本：验证 ReaderLab 没有重新滑回 Skill 专项。
