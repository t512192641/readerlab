# Decision Archive

这里保存已被收窄、替代或只适用于旧阶段的历史决策全文。普通启动不需要读取本文件；只有处理旧 Skill 包页面生成、旧翻译/精读产物、旧四段式页面或旧质量门兼容时，才按需回看。

## D-001 Skill 正文只翻译 `SKILL.md`（旧 Skill 包实现约束，已被 D-014/D-015 收窄）

Skill 正文区域只允许出现源 `SKILL.md` 的中文化内容。字段名、命令、路径、触发词、代码块等必要技术标识可以保留英文。

讲解、机制解释、使用建议和 ReaderLab 旁批不能写进 Skill 正文。

该决策只约束当前旧版 Skill 包生成器中的“Skill 正文”字段，不能再被解释为“主阅读页必须承载完整执行视图”。复杂材料阅读化的新主规则见 D-014 和 D-015。

## D-002 关联材料单独展示

一个 Skill 的完整功能不只由 `SKILL.md` 构成。模板、脚本、参考文件和约束文件如果影响理解，应在“关联材料”区域单独说明。

关联材料说明必须标明来源文件，不能混进 Skill 正文。

## D-002a 关联材料必须 inventory-first（扫描仍保留，读者页输出已收窄）

ReaderLab 不能靠人工枚举补漏。生成阅读页前必须先扫描每个 Skill 的构成材料，包括 `SKILL.md.tmpl`、`bin/`、`scripts/`、`sections/`、`templates/`、`docs/`、`src/`、测试参考和 hook command 引用。

扫描结果必须进入 manifest、Codex 吸收层或审计层，保证可追溯。读者页的“关联说明”不再机械输出“机制盘点”和“执行机制”：只有 hooks、脚本、模板、状态文件、外部依赖或运行约束确实影响理解时才进入主阅读页；没有真实关联价值时不造空说明。

## D-003 未完成 Skill 不生成占位阅读页

未完成 Skill 只在 `04_主控清单.md` 和 `manifest.json` 中保留目标路径、状态和覆盖信息。

不要生成“这里必须替换为完整中文译文”这类占位页，避免在 Obsidian 里混淆视线。

## D-004 重点与亮点不强行凑数（旧规则，已被 D-023 下线）

重点与亮点数量可以是 0-3 条。没有真正值得标的内容时，不为了版式完整强行制造高亮。

重点来源可以是 Skill 正文或关联材料，但必须清楚标明来源。

该决策只解释旧 `highlights` 字段为什么不应强行凑数。用户已确认当前 `重点与亮点` 功能下线；新生成器不应继续默认输出该模块。

## D-005 用代表性样本集打稳四段式标准

四段式标准不只用 `freeze + unfreeze` 两个短 Skill 验证，而是用小规模代表性样本集验证：短页、带脚本页、表格页、组合型页和较长流程页。

当前样本集为 `unfreeze`、`freeze`、`careful`、`guard`、`gstack-upgrade`。继续扩样本时，优先选中等长度且结构不同的 Skill；不要直接进入 `spec`、`office-hours`、`autoplan` 这类超大 Skill，也不要扩大成整包批量翻译。

## D-006 公共术语解释前置并去重

术语解释是正文阅读前的钥匙，不应藏在正文之后的关联材料里。

ReaderLab 应生成包级公共术语页，例如 `02_Skill阅读术语表.md`。单个 Skill 页在“短导读”末尾放术语表链接和本页重点术语；关联材料不再重复整段术语解释，只保留机制盘点和执行机制。

## D-007 人工样本不等于自动精读能力

当前 5 个完成页是已验收打样页：正文译文、导读和旧重点旁批主要由人工维护，用来定义早期成品形态和验收标准。

ReaderLab 的自动化部分目前是扫描、组织、术语链接、机制盘点、manifest 和验证。后续精读、翻译和读者辅助可以交给专用 Agent/Skill 工作流，再由 ReaderLab 校验产物；但旧 `重点与亮点` 不再作为默认读者辅助模块。

## D-008 Agent 精读产物使用外部 JSON 契约

新的完成页或需复核页不应继续写入 `COMPLETED_SKILL_TRANSLATIONS` 扩大 Python 字典。

Agent 精读产物应使用外部 JSON 文件，当前 schema 为 `readerlab.skill-reading.v1`，默认位置为 `data/skill-readings/<book-id>/<skill>.json`。ReaderLab 读取这些产物生成页面，并在 manifest 中记录 `reading_source=agent_reading` 和产物路径。

Agent 产物可以生成 `needs_review` 页面；`needs_review` 表示“有页面但覆盖指标不足”，不能计入完成，也不能显示成待生成。

## D-009 Agent 正文译文不得摘要化（证据覆盖约束，非主阅读页形态约束）

Agent 精读产物中的 `skill_body_translation` 必须是 `SKILL.md` 的完整中文译文，而不是摘要、压缩解读或教程化改写。

导读、机制解释和来源证据必须分别进入 `guide`、`related_materials_explanation` 和 `evidence`。旧 `highlights` 字段只作为历史产物兼容，不再作为默认生成要求；这些辅助内容都不能补正文覆盖率。

长 Skill 应采用分块翻译和覆盖清单，确保标题、段落和代码块数量被保留；如果无法完整覆盖，应保持 `needs_review`，不能标记为完成。

该决策用于防止信息流失，仍然约束“证据附录 / 完整译文 / 覆盖校验”。它不要求主阅读页机械展示所有代码、运行外壳和执行细节。

## D-010 分块精读由 ReaderLab 掌控状态

长 Skill 的完整翻译不能依赖 Agent 自觉一次性循环处理。

ReaderLab 或专用 Skill 工作流必须负责切块、记录 block id/行号/hash、分派翻译、合并结果、校验覆盖和驱动缺块重试。Agent 只负责有限 block 的翻译和解释。

分块产物缺失、源 hash 不匹配、标题/段落/代码块覆盖不足时，页面最多为 `needs_review`，不能进入 `completed`。

## D-011 分块精读必须记录时间和可得消耗

从下一轮真实 Skill 翻译开始，每个 Skill 和每个 part 都必须记录开始时间、结束时间、耗时、block 范围、block 数、负责子 Agent 和结果状态。

如果当前工具能提供 token 使用量，则记录 token；如果不能提供，则明确记录 `token_usage_unavailable`，不能留空或假估。

这些指标用于后续估算“解读一个 Skill”的时间和成本，也用于比较小块拆分、多 Agent 并行和不同长度 Skill 的真实效率。

## D-012 稳定性测试应批量覆盖不同长度和类型

单个 Skill 跑通只能证明局部可行，不能证明流水线稳定。

后续测试应一次选择 3-4 个不同类型、不同长度的未完成 Skill，形成小 / 中 / 较长组合，再比较 completed 率、返工率、耗时、质量问题类型和人工验收压力。

已完成的 `benchmark` / `benchmark-models` 不再作为新闭环证明重复使用。

## D-013 质量门必须覆盖短英文残留和代码注释残留

大段英文比例检查不够。Agent 可能留下短英文指令句、提示句、半翻译混合句，或在代码块中保留英文说明注释；这些都会影响中文阅读页质量。

ReaderLab 的 completed 判断必须把这类问题纳入 block issues。允许保留命令、路径、flag、变量、状态值、工具名、模型名、CLI 输出字符串、code span 和明确标注的原始英文触发示例；但普通说明文字、用户提示文案和代码注释中的自然语言应中文化。

如果这类问题命中，页面保持 `needs_review`，不能因为 block 覆盖完整而进入 `completed`。
