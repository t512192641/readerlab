# Technical Lead Design Asset Method

## 目的

技术负责人 / 设计资产提炼层用于处理 Skill 包、代码文档、Agent 工作流、Prompt 系统、工具链和工程设计资料。

它面向的主要读者是产品负责人、需求设计者和主要用户，而不是全职工程师。它不是解释术语，也不是夸设计精妙，而是由 Codex 替读者看出其不容易感知的技术和工程设计，并翻译成可复用的产品 / 方法资产。

它必须帮助读者理解：

- 为什么这样设计。
- 它解决什么问题。
- 它约束什么行为。
- 它防止什么失败。
- 它带来什么代价。
- 它能否迁移。
- 它能沉淀成什么设计卡片。

## Reader Role

默认用户不知道自己不知道什么。ReaderLab 不应等待用户指出“这里有什么技术精妙之处”，而应主动识别：

- 这里是否有强用户意图设计。
- 这里是否有成熟的问题建模。
- 这里是否有输入治理、输出契约或失败路径设计。
- 这里是否有防止 Agent 跑偏的约束。
- 这里是否有可迁移的产品方法或工作流模式。
- 这里是否有不能照搬的上下文依赖。

输出必须使用产品负责人能使用的语言，而不是把工程术语原样抛给读者。

## Design Atom Analysis

每个技术模块或 Skill 至少从以下设计原子中选择相关项分析：

```text
trigger
user_intent
inputs
outputs
workflow
tools
state
constraints
failure_modes
human_gate
output_contract
verification
risk_control
transferable_pattern
non_transferable_part
```

不是每个模块都要强行填满所有原子。缺失但重要的原子应进入风险或待复核项。面向读者展示时，不必暴露这些字段名；字段是 Codex 的分析骨架，读者页应翻译成自然语言。

## 四个问题

每个设计原子都应回答：

1. 它解决什么问题。
2. 它为什么放在这里。
3. 它带来什么好处和代价。
4. 它能不能迁移到别的 Skill / 工作流。
5. 它对产品负责人有什么可复用价值。

## 输出层级

### 技术负责人讲解

放在净化正文之后或附近，用成段语言讲清这个 Skill / 技术材料的设计思路。它回答“这个东西为什么这样设计”，而不是把正文拆成许多碎片栏目。

### Capability Map

用于 Skill 包或工程材料的能力地图，必须包含：

- trigger signals
- near-neighbor exclusions
- method atoms
- required inputs
- output contract
- verification
- route decisions
- source refs
- confidence
- review status

### Design Asset Notes

用于更完整的技术判断，尤其是跨模块设计取舍、风险和可迁移模式。它的最终读者不是工程师复盘代码，而是产品负责人沉淀方法资产。

最低应包含：

```text
设计模式名称
适用场景
解决的问题
原材料中如何体现
可复用做法
使用前提
失败风险
什么时候不要用
```

### 设计资产卡输出契约

设计资产卡不是摘要，也不是术语解释。每张卡必须面向产品负责人回答：

- 适用场景：什么问题、材料类型或工作流中可以考虑使用。
- 解决问题：它实际约束了什么行为，降低了什么失败概率。
- 原材料依据：来自材料中的显性说明，还是 Codex 基于结构做出的工程推断。
- 可复用做法：下次复用时可以复制的设计动作。
- 使用前提：需要什么输入质量、组织条件、人类判断或工具能力。
- 失败风险：照搬时最容易坏在哪里。
- 什么时候不要用：哪些场景下它会增加复杂度、制造误判或压错目标。

## 必须允许负面判断

技术负责人 / 设计资产提炼层必须能说：

- 这是核心设计。
- 这是历史包袱。
- 这是模板展示，不是能力核心。
- 这是为了规避 Agent 风险。
- 这个字段有用，但约束不足。
- 这个流程可迁移，但依赖上下文。
- 这个设计不应迁移。

## 失败条件

出现以下情况，本层输出失败：

- 只复述 Skill 文档。
- 只解释技术名词。
- 全是正面夸奖，没有代价和边界。
- 把安装命令当核心能力。
- 把模板字段当方法论，但不能说明字段约束了什么行为。
- 不说明什么时候不要用。
- 不区分原文显性说明和 AI 工程推断。
- 只从工程师视角解释实现，不说明对产品负责人有什么可复用价值。
- 没有沉淀可复用设计卡片。

## 与一手正文的关系

技术分析不能污染一手正文。Skill / 工程材料的净化正文保留材料主体；设计资产提炼层解释设计取舍、隐藏工程约束和迁移价值。完整命令、脚本、hash、路径和执行证据默认进入 audit、附录或技术解析，不进入正文主体。
