# 结构化提炼评估

## 旧输出诊断

共同失败原因不是“没有结构”，而是结构化提炼没有形成同一条可验证链路：全书主问题、章节/部件功能、重点/亮点选取理由、来源精度和降级边界没有闭合。

### `readerlab-fullbook-demo-v0`

- 全书主线能把部件串成“目标 -> 思维 -> 执行 -> 公司案例 -> 人类未来”，但更像目录重述和主题归纳。
- 章节/部件路线基本还是“按部件读”，没有拆出每个部件内部的主干、支撑、案例、噪音。
- 亮点提炼有判断，但 refs 主要是 spine 级，无法判断亮点来自具体段落，还是由标题和既有印象推出来。
- 自身已承认 refs 不是精确段落或 char span。

### `readerlab-elon-full-product-v1`

- v1 的强项是把主稿收敛成机制链，并加入通过/降级/拒绝。
- 但全书地图和章节/部件地图不显式，结构判断折进连续主稿和筛选表，难以评审每个章节承担什么结构功能。
- 候选筛选方向正确，但候选池生成过程仍不够透明，容易变成“人工挑好点再补理由”。
- 来源边界仍偏弱：主要继承 v0 location map，不是 fresh source-grounded contract。

### `readerlab-elon-fragment-capability-eval-v0`

- 片段卡在局部误读防护上最强，能给出问题、动作、反例、审查标准。
- 但它主动放弃全书总纲，不能证明全书 Top 选点能力。
- 8 个片段覆盖不同难点，但选择依据仍是手工评测口径，不是从全书结构中推出的必选节点。
- 它能证明片段深读方向有价值，不能证明结构化提炼稳定生产链。

## 三个粒度的共同问题

| 粒度 | 旧问题 | 本轮改法 |
|---|---|---|
| 全书地图 | 容易把目录层级改写成主线，缺少主问题推进和主干/支撑/案例/噪音判断。 | 先定材料性质和主问题，再表达约束传导链。 |
| 章节/部件地图 | 要么只有部件级路线，要么散落在主稿里。 | 选部件 II 做内部结构角色图，并说明它如何承接/传导全书机制。 |
| 重点/亮点提炼 | 容易混合金句、旁批、价值提炼和候选洞察。 | 每个亮点必须说明入选理由、降级/拒绝理由、来源范围和边界。 |

## 结构化提炼仿型能力矩阵

| 仿型 | 定性 | 筛选 | 排序 | 组织 | 边界 | 表达 |
|---|---|---|---|---|---|---|
| 仓颉 | 替代方案：类型/术语/方法原子识别 | 立即学习：候选池、通过/降级/拒绝 | 立即学习：V1/V2/V3 后再排优先级 | 替代方案：框架/原则/案例/反例/术语分池 | 立即学习：降级但保留、拒绝升格 | 不吸收：不要把候选池直接露给读者 |
| 李继刚式深读 | 立即学习：缺口、增量、不增量 | 替代方案：按认知增量筛掉浅摘要 | 替代方案：按机制传导排主线 | 立即学习：从目录改成约束传导链 | 立即学习：书内支撑/解释者推断/外部待核验 | 立即学习：让读者完成认知旅程 |
| source-grounded / NotebookLM 类 | 不吸收：不负责内容定性 | 替代方案：无来源项直接降权 | 替代方案：按证据精度排序置信度 | 未来储备：更强 source package / 引用系统 | 立即学习：source_scope、coverage、location refs | 替代方案：读者侧少露审计，审计侧可复核 |
| ReaderLab 现有能力 | 替代方案：已有全书问题和 reader_gain，但偏浅 | 未来储备：v1 有候选审计，但未机制化 | 未来储备：能列结构，缺稳定选点理由 | 立即学习：reader/audit 分离、LifeAtlas 包装 | 立即学习：human_status pending、validator 不等于质量 | 替代方案：v1 主稿方向可保留，v0 短页不保留 |

主 Agent 采纳顺序：

1. 李继刚先定性：材料性质、核心缺口、增量/不增量、机制链。
2. 仓颉再筛选：候选池、通过/降级/拒绝、选点理由。
3. source-grounded 加约束：每个主张绑定 source scope、location refs、coverage 和 pending 状态。
4. ReaderLab 最后表达：读者侧输出连续可读主线、部件地图、亮点；审计侧保留候选和证据。

## 新增护栏：局部抽取必须审查上下游依赖与组合增益

用户指出一个关键风险：某个 Skill 或仿型的单一环节，可能是为下一环服务的；脱离原方法链路后，该环节本身可能失效，也可能让后续输出质量下降。这个风险成立。

用户进一步指出：问题不只是“不破坏上下游”，还要证明抽出的几个部分加在一起能不能产生 1+1>2 的效果。这个判断也成立。多个优秀环节如果没有共同输入、共同质量门和共同输出目标，组合后可能只是外观拼接，甚至降低质量。

因此，本报告里的“李继刚负责定性、仓颉负责筛选、source-grounded 负责来源边界”只能视为本轮手工重考中成立的工作假设，不能直接升级成稳定生产规则。

后续机制化前必须补一层依赖与组合增益审查：

| 拟抽取环节 | 需检查的依赖 | 当前判断 |
|---|---|---|
| 李继刚式深读的缺口/增量/机制传导 | 是否依赖完整长文深读、认知旅程和证据分层；只抽“缺口”会不会变成漂亮开场。 | 需要继续审查；不能只保留标题式缺口。 |
| 仓颉的候选池和通过/降级/拒绝 | 是否依赖 Stage 0 整书理解和 V1/V2/V3 验证；没有前置候选生成时会不会变成人工补理由。 | 强依赖上下游；模板必须保留候选来源和验证门。 |
| source-grounded 的来源边界 | 是否只提供证据约束，而不负责内容定性；如果放在最后，会不会只能补 caveat，不能纠正前面误判。 | 应前后双向使用：前置约束候选，后置标明边界。 |

组合增益必须回答：

- 李继刚的“缺口/机制”是否真正改善仓颉候选池的生成质量，而不只是提供开场措辞。
- 仓颉的“通过/降级/拒绝”是否真正提高李继刚机制链的选点精度，而不只是把已有判断表格化。
- source-grounded 是否前置约束候选生成并后置校验结论，而不只是最后补 caveat。
- 三者合并后，是否比任一单一路线更能减少目录复述、解释结构传导、说明选点理由和保留边界。

下一轮验证方式必须改成同题多路线对照：

| 路线 | 生成者 | 作用 |
|---|---|---|
| 仓颉-only | 独立 worker | 检查候选池、通过/降级/拒绝单独使用时的效果。 |
| 李继刚-only | 独立 worker | 检查缺口、增量、机制传导单独使用时的效果。 |
| ReaderLab 当前链路 | 独立 worker | 作为本轮组合假设的复现版本。 |
| 组合路线 | 独立 worker | 测试“李继刚定性 + 仓颉筛选 + source-grounded 边界”是否有组合增益。 |
| 评审路线 | 独立 worker | 不生成内容，只比较上述路线。 |

所有路线必须使用同一材料、同一材料范围、同一输出契约。评审必须比较目录复述、缺口说明、结构传导、选点理由、不升格理由、来源边界和读者可用性。主 Agent 最后必须给出倾向性处理方案：组合胜出才进入生产链；组合不胜出时，选择最强单一路线作为主方法，其他仿型降级为辅助 gate、储备或不吸收。

新的吸收规则：

- 可独立抽取：只有当该环节输入、输出、判断标准和质量门都能在 ReaderLab 中被等价承接。
- 需连同上下游抽取：当该环节依赖原方法前置理解或后续验证。
- 不应抽取：当该环节离开原方法后只剩字段外观、术语或风格。
- 组合无增益：当多个环节只是并排存在，没有产生更强判断链；这种情况不能进入生产链。

下一轮不能只验证“拼接后的页面是否更顺”，还要验证这些环节之间的数据和判断是否真的接上，以及组合后是否优于单一路线。

## Meta Skill 裁判

Meta Skill 评估只作为裁判视角，不作为 ReaderLab 阅读输出仿型。本轮读取并应用了：

- `skill-engineering-method.md`
- `output-eval-method.md`

裁判结论：

- 结构化提炼应定义为 ReaderLab 的“主干判断课”，不是字段生成课，也不是整套阅读产品的总包。
- 参考扫描应收窄在 3-5 个对象，只抽取能通过复用、生成力、独特性、边界清晰的模式。
- 旧输出仍有“只奖励字段存在”的风险：catalog map、coverage、assertions、候选字段齐全都不能证明真实提炼质量。
- 现在应该采用小范围重考、source/location、候选池、拒绝/降级、新旧对比和 forbidden checks。
- provider-backed output eval、blind A/B、正式 JSON schema、生成器改造、多材料扩测都应后置。
- 新增：仿型环节依赖与组合增益审查必须前置于模板化，避免把“局部借鉴”误判成可靠模块化。

主 Agent 采纳：

- 采用：小范围重考、候选池、降级/拒绝、来源范围、新旧对比。
- 储备：blind A/B、provider-backed eval、多材料扩测。
- 不吸收：把 Meta Skill 当阅读模板，或把 CTK/MEM/Meta Skill 放进内容仿型清单。

## Output Risk

- 泛泛摘要：把“讲了什么”冒充“补了什么缺口”。
- 目录冒充地图：列出目标、工程、组织、制造、文明风险，但不解释约束传导。
- 低覆盖冒充全书：继承旧 refs 或只读局部，却写成全书结论。
- 精华无来源：亮点、洞察、候选能力没有 location/source refs。
- 候选缺失：只展示胜出项，不展示降级和拒绝。
- 金句升格：把“硬核工作”“多行星”“共情力不是资产”等高风险表达包装成通用原则。
- 审计污染正文：机器状态、hash、validator、coverage 明细进入读者页。
- 读者收益不明：地图有结构，但读者不知道如何理解、批注、复核或迁移。

## 新旧对比

| 检查项 | 旧输出 | 本轮小范围重考 |
|---|---|---|
| 是否减少目录复述 | v0/fullbook 仍像部件目录改写。 | 用主问题和约束传导重写全书主线。 |
| 是否说明书补了什么缺口 | v1 有改善，v0 不足。 | 明确补“宏大目标如何压成工程/组织/制造/公司行动”的理解缺口。 |
| 是否解释结构如何传导 | v0 列链条，传导解释不足。 | 明确部件 II 如何承接部件 I 并传导到部件 III。 |
| 是否说明选点和不升格理由 | v1 有候选筛选，fragment 有误读防护，但分散。 | 候选池、通过/降级/拒绝和亮点理由进入同一条链。 |
| 是否保留来源和人工边界 | 都保留边界，但多为继承 refs。 | 继续保留 inherited refs caveat，并明确 fresh contract 未完成。 |

## 最终判断

本轮最小闭环比旧输出更能说明主问题、机制传导、选点理由和不升格理由。它证明了一条手工结构化提炼生产链值得继续，但没有证明 ReaderLab 生成器已掌握该能力，也没有证明全书产品 ready。
