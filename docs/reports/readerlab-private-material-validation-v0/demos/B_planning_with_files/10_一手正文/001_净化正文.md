# planning-with-files 净化正文

## 这份材料要解决什么问题

`planning-with-files` 面向复杂、多步骤、跨多次工具调用的任务。它的核心判断是：上下文窗口像临时内存，容易丢失；文件系统像持久磁盘，应该承接任务目标、发现、错误和阶段进度。

它不是一个让人“多写计划”的泛泛建议，而是一套把复杂任务外化到项目目录里的工作协议。它要求在任务开始时建立三个文件：`task_plan.md` 记录阶段、状态和决策，`findings.md` 记录研究发现，`progress.md` 记录过程、测试和错误。

## 触发条件

当任务需要三步以上、研究过程、项目构建、较多工具调用，或可能跨会话继续时，应启用这套方法。简单问答、单文件小改、快速查询不需要启用。

材料还强调会话恢复：开始工作前先检查是否存在上一次遗留的未同步上下文；如果有，就先看实际变更、读取已有计划文件，并把恢复信息并入当前工作状态。

## 核心流程

第一步是先建立计划文件，再开始复杂任务。计划文件要记录目标、阶段、状态和已知约束，避免 agent 在后续操作里忘记原始目标。

第二步是把发现写入文件。材料提出一个强规则：每进行两次查看、浏览或搜索操作，就要把关键发现保存下来。这个规则的目标不是形式主义，而是防止观察到的信息在上下文压缩或窗口切换后丢失。

第三步是在做重要决策前重读计划。这个动作让当前判断重新对齐任务目标，降低“越做越偏”的风险。

第四步是阶段结束后更新状态。完成一个阶段时，需要把状态从进行中改为完成，并记录创建或修改了哪些文件、遇到了哪些问题、做过哪些验证。

第五步是记录所有错误。错误记录不是复盘装饰，而是防止重复失败的机制。材料明确要求同一个失败不能原样重试，要诊断、换方法，三次后升级给用户。

## 输出要求

这套方法的标准输出不是一个最终答案，而是一组可恢复的工作记忆文件。`task_plan.md` 告诉后续执行者“我在哪、要去哪、下一步是什么”；`findings.md` 保存关键事实；`progress.md` 保存已经做过什么和验证结果。

## 边界

这份 Skill 的运行外壳包含 hooks、shell 命令、跨平台脚本路径和 Claude 插件目录约定。这些内容对真实插件执行有用，但对 ReaderLab 的读者正文不是主线。正文应该保留“为什么要把工作记忆写入文件、什么时候启用、怎样更新、怎样防止重复失败”，而不是让命令片段占据阅读中心。

它也不能被直接迁移成 ReaderLab 的正式 Skill。可迁移的是若干设计原则：复杂任务需要外部化状态；发现要及时落盘；决策前要重新对齐目标；错误要变成下一步策略的输入。是否生成正式 Skill，还需要独立的触发、输入、步骤、输出、边界和证据判断。

## 设计亮点

第一，材料把“记忆不足”从模型缺陷转成流程设计问题。它不假设 agent 会自然记住所有上下文，而是要求关键状态落到项目文件。

第二，材料把恢复能力放在开始阶段，而不是等出错后补救。会话恢复检查让跨会话工作有一个入口。

第三，材料把错误管理写成协议。三次失败后升级、不能原样重试、每次失败都要记录，这使复杂任务更容易被审计和接手。

第四，材料明确区分适用和不适用场景。它不要求所有任务都创建计划文件，这一点避免了方法本身变成负担。
