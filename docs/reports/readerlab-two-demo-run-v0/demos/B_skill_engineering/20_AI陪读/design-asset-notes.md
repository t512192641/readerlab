# Design Asset Notes: gstack spec

## 设计模式一：决策型提问

适用场景：用户的输入还停留在想法、愿望、抱怨或模糊需求，需要先变成可执行规格。

解决问题：防止 agent 把“用户还没决定”误处理成“可以继续执行”。它约束 agent 必须说明正在决定什么、为什么重要、推荐什么、各选项代价是什么。

原材料中如何体现：材料要求每个选择都包含问题解释、选错代价、推荐理由、完整度、利弊和最终取舍。

可复用做法：在 ReaderLab、LifeAtlas 或其他工作流中，凡是会改变范围、产物形状、数据归属、发布边界的地方，都把“选择题”升级成“决策说明”。

使用前提：问题本身有真实分叉，且用户的判断会改变后续结果。

失败风险：如果每个小问题都套完整决策说明，会拖慢流程，让用户疲劳。

什么时候不要用：事实查询、低风险格式选择、已经有明确用户指令的执行步骤，不需要强行升级成决策问题。

依据类型：原文显性说明为主，产品复用价值为工程推断。

## 设计模式二：触发边界先行

适用场景：一套工具或 agent 系统中有多个相邻能力，容易互相抢任务。

解决问题：防止一个流程因为“看起来也能做”而吞掉不属于自己的任务。`spec` 只负责把需求整理成规格，不负责所有计划、测试、审查、发布和上下文保存。

原材料中如何体现：材料列出 `spec` 的触发短语，也列出产品想法、策略、架构、测试、审查、发布等相邻路由。

可复用做法：每个 ReaderLab 子能力都应写清楚“什么时候用”和“什么时候转交”。例如高阶讲解、正文净化、设计资产提炼和审计合同不能互相冒充。

使用前提：系统里确实存在多个能力入口，否则边界规则会显得过度。

失败风险：边界写得太细会制造路由成本；边界写得太宽又会让产物失真。

什么时候不要用：一次性小工具、只有一个明确动作的脚本，不需要复杂路由。

依据类型：原文显性说明和工程推断混合。

## 设计模式三：证据先于规格

适用场景：用户要把需求写成 issue、ticket 或执行合同，但问题涉及真实代码、系统或已有流程。

解决问题：防止规格只根据对话想象生成，导致执行者打开代码后发现关键约束不存在或完全不同。

原材料中如何体现：材料要求进入技术追问前必须先读至少一条代码库证据，并在问题里引用具体文件、结构或搜索结果。

可复用做法：ReaderLab 的工程材料陪读也可以采用同一原则：先确认正文或源材料里真的有什么，再写高阶讲解、设计资产或候选沉淀。

使用前提：存在可读取的代码、文档、结构或事实层。

失败风险：为了“找证据”而扩大读取范围，反而拖慢任务或读到无关历史。

什么时候不要用：纯创意探索、尚未进入执行合同的产品讨论，不需要过早要求代码证据。

依据类型：原文显性说明为主。

## 设计模式四：可交接规格闭环

适用场景：需求不是写给当前 agent 自己做，而是要进入 backlog、GitHub issue、另一位工程师或子 agent。

解决问题：把“理解了”变成“可交接”。它要求规格包含现状、变更、验收、测试、回滚、工作量、文件参考和不做范围。

原材料中如何体现：材料给出 issue 模板、验收标准、测试计划、回滚计划、归档和可选 spawn 交接流程。

可复用做法：ReaderLab 后续如果把阅读结论沉淀成可执行改造任务，也应保留“当前事实、提议改动、验收方式、回滚边界”四件套。

使用前提：工作已经过了“是否值得做”的阶段，准备进入执行。

失败风险：过早写成 issue 会冻结错误问题；过度模板化会让小任务成本变高。

什么时候不要用：用户还在探索方向，或者材料只是用于阅读理解而非交付执行时，不应强行 issue 化。

依据类型：原文显性说明为主，ReaderLab 迁移价值为工程推断。

## 设计模式五：环境感知但不让环境抢主线

适用场景：同一个流程会运行在交互会话、无人值守会话、子 agent、不同宿主工具或不同项目状态中。

解决问题：不同环境下同一动作的风险不同。有人能回答时可以问；没人能回答时应阻断；子会话里可以自动选推荐项，但要报告。

原材料中如何体现：材料区分 interactive、headless、spawned、Conductor、plan mode，并为工具失败、无人值守和子会话给出不同处理。

可复用做法：ReaderLab 后续如果引入 writer、reader、controller 多角色，也应区分“能向用户确认的主控”和“只能执行并报告的子 agent”。

使用前提：工作流确实跨环境运行，并且环境差异会改变安全边界。

失败风险：环境探测代码和宿主兼容说明会迅速膨胀，污染读者对主体方法的理解。

什么时候不要用：纯阅读产物、稳定单机脚本、不会跨宿主执行的流程，不应引入复杂环境分支。

依据类型：原文显性说明为主。

## 设计模式六：运行外壳隔离

适用场景：材料同时包含主体方法和大量安装、升级、同步、遥测、路径、状态检查等机器协议。

解决问题：防止读者把工具维护细节误读成方法论主体。对 ReaderLab 来说，这决定了净化正文是否能保住主线。

原材料中如何体现：材料的大量内容用于升级提示、首次运行提示、路由注入、vendored 迁移、artifact 同步和遥测处理，但这些不改变 `spec` 的核心目的。

可复用做法：把“读者必须理解的主体流程”和“机器必须执行的协议”分层展示。正文讲方法，审计层保留剥离类别和风险。

使用前提：原材料确实是工程/Skill 文档，而不是普通文章。

失败风险：剥离过度会丢掉重要约束；剥离不足会让正文变成命令手册。

什么时候不要用：源材料本身就是命令参考或操作手册，且读者目标就是执行命令时，不能强行剥离到只剩理念。

依据类型：ReaderLab 对材料结构的工程推断。
