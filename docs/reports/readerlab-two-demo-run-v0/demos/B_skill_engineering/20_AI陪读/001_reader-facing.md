# 读这份 `spec`，重点不是命令，而是它怎样保护需求不变形

这份材料表面上很重，因为里面塞了大量环境检查、一次性提示、同步规则和工具调用格式。真正值得读的主线不是这些命令，而是一个更大的问题：当用户只给出一个含糊想法时，agent 怎样把它变成别人能执行的规格，同时不偷偷替用户做决定。

`spec` 的核心能力不是“会写 issue”。会写 issue 只是结果。它真正解决的是需求进入工程系统前最常见的变形：范围没有说清楚，风险没人承认，选项被压扁，用户其实还没决定，agent 却已经往下做了。它先问清谁受影响、现状是什么、目标是什么、为什么现在做、怎样算完成；再锁定边界、最小版本、失败模式和回滚方式。这样做的目的，是让后续执行者拿到的是一个可执行问题，而不是一团顺滑但仍然含糊的描述。

你可以把它看成一个需求入口的安检台。普通做法会把用户的话整理成一段顺滑文本；这个 Skill 更关心“这段文本后面能不能真的执行”。所以它要求进入技术追问前先读代码或项目证据，不能靠想象提问；要求草稿给用户复核，不能第一轮就把 issue 扔进 backlog；还要求质量检查和脱敏检查，避免规格虽然能执行，却把敏感内容带进下游。

它的交付闭环也很清楚：最终不是只给一段建议，而是形成 backlog-ready issue、本地归档，并在合适场景下把工作交给独立执行环境。这个设计说明 `spec` 的目标不是“写作”，而是把需求从口头意图推到可交接的执行物。

最有价值的一点，是它把“问用户”做成了结构化产品设计。一个好问题不能只是“要不要”。它要告诉用户现在在决定什么，为什么这个决定重要，选错会损失什么，推荐哪个选项，以及每个选项完整到什么程度。这样用户不是被迫从两个按钮里猜，而是在理解代价后做选择。

同时，这份材料也暴露了一个必须降级处理的部分：大量升级、遥测、路由、同步和宿主兼容规则，对工具运行很重要，但对读者理解 `spec` 的主体方法并不重要。如果把这些都原样放进阅读正文，读者会误以为核心能力是配置一套工具，而不是把模糊需求变成可执行规格。ReaderLab 在这里应当保留它们背后的设计意图，把命令和机器状态移到审计层。

这份材料能迁移的，是四种方法。第一，把入口触发条件说清楚，避免一个流程吞掉所有任务。第二，把关键选择显性化，让推荐和风险一起出现。第三，让规格生成先受事实约束，再进入方案描述。第四，为不同执行环境设置不同推进方式：有人能回答就问，没人能回答就停，子任务执行中可以按默认推荐推进但必须报告。

不能直接照搬的，是它的工具栈外壳。比如 Claude Code、Conductor、gstack 的本地目录、遥测和 artifact 同步规则，都高度依赖原项目环境。它们适合当作“复杂工具如何处理长期状态”的案例，不应变成 ReaderLab 或 LifeAtlas 的默认工作流。

所以读这份 `spec` 时，最该吸收的是需求入口的治理方式；需要降级的是一次性配置和宿主兼容细节；应该拒绝的是把所有环境噪音都当成核心方法。真正的收获是：一个好的规格流程，不只是帮你把想法写得更完整，还要保护你的真实决定不被自动化吞掉。
