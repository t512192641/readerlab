# gstack spec：净化正文

`spec` 是 gstack 中用于把含糊需求整理成可执行规格的 Skill。它适合在用户说“把这个想法写成 issue”“整理成 ticket”“做成 backlog item”或“帮我 spec this out”时启动。它的目标不是马上写代码，而是把意图、范围、约束、风险和交付形状整理到足够明确，让后续 agent、工程师或 `/ship` 流程能接住。

这份材料的显性用途有三层。第一层是“立项”：把一句模糊请求转成一个可以被跟踪、讨论、执行和关闭的事项。第二层是“分流”：在需要时可以把工作交给新的执行环境或后续流程，但它先要把问题讲清楚。第三层是“闭环”：规格不是孤立文档，而是会和 issue、分支、后续交付、合并关闭源 issue 等动作连接。

它的触发条件很具体：用户表达的是整理需求、提交 issue、写 ticket、转 backlog，而不是排查 bug、做代码审查、视觉打磨或发布上线。材料中特意列出相邻但不同的工作类型，例如产品想法、策略范围、架构、设计评审、测试、代码审查、发布和保存上下文。这说明 `spec` 的边界意识很强：它负责把需求变清楚，不负责代替所有工作流。

用户意图被理解为“我现在还没有一个可执行问题”。因此这个 Skill 要先判断项目处境，再帮助用户把问题落到规格。它关注的不是“写一段漂亮描述”，而是“后面的人能不能照着做”。所以它会强调决策、完整性、推荐理由、取舍、风险、停止点和明确回复，而不是只输出一般建议。

核心流程可以读成两层：一层是 gstack 的运行外壳，一层是 `/spec` 真正的规格生产流程。读者应先抓住第二层。

规格生产流程从“为什么”开始。它要求先问清五件事：谁受影响、当前真实行为是什么、期望行为是什么、为什么现在要做、怎样才算完成。材料明确禁止第一轮就产出 issue，因为含糊工作一旦进入 backlog，后面会把不确定性转嫁给执行者。

第二步是锁范围和边界。它要问清什么明确不做、会触碰哪些系统、有没有顺序约束、最小可交付版本是什么、失败模式和回滚方式是什么。这一段的产品意义是防止 scope creep，也防止把一个应该拆分的工作塞进单个 issue。

第三步是技术追问，而且必须先读代码或项目证据。材料要求在进入技术问题前，至少通过搜索、文件列表或阅读拿到一条实际证据，再基于证据追问数据模型、API、后台任务、UI、基础设施和测试问题。这里的设计重点是：规格不能只来自对话想象，必须被当前代码或项目结构约束。

第四步是草稿复核。它要求先给出完整 issue 草稿，再问用户“是否准确捕捉了你想要的东西，哪里错了”，并迭代到用户确认。这一步把规格看成双方确认的执行合同，而不是 agent 单方面生成的文档。

第五步是质量和风险门。材料设计了第二模型质量检查、语义脱敏检查、正则脱敏检查，并明确高风险内容不能继续进入下游。它还把 `--no-gate` 的作用限制在质量评分上，不能跳过脱敏。这说明它把“规格可执行”和“规格可公开/可归档”分成两个不同问题。

第六步是落地和交接。规格最终会被写成 GitHub issue，并在本地归档；在执行模式下还可以创建独立 worktree 并把规格交给子 agent。材料要求处理 dirty worktree、SHA pin、唯一分支名、最终确认和 stash 恢复说明。这些属于规格流的交接机制，不是读者正文的核心，但能说明它追求的是从模糊意图到可执行工作的闭环。

运行外壳可以再读成五段。

第一段是运行前定位。材料会检查当前项目、分支、会话类型、是否已有路由规则、是否首次使用、是否有历史学习记录、是否处于计划模式，以及是否有可用的外部知识或同步状态。这些信息本身不是规格内容，但它们决定了 Skill 该以什么强度介入：直接继续、提示用户、阻断、自动选择推荐项，还是把决策写成可回答的问题。

第二段是一次性环境引导。材料包含升级、写作风格、遥测、主动建议、项目路由、vendored 迁移、artifact 同步等提示。这些不是 `spec` 的主体能力，而是 gstack 作为一套持续工具栈在维护长期使用体验。对 ReaderLab 来说，这些应被剥离出正文主体，只保留它们背后的设计意图：首次使用要降低迷路概率，长期使用要减少重复配置，涉及隐私或项目规则时要让用户明确选择。

第三段是决策提问规范。材料要求每个用户决策都不是一句“要不要”，而要包含问题本身、为什么重要、选错会怎样、推荐选项、各选项完整度、利弊、最终取舍。它还规定了交互失败时的处理方式：如果工具不可用，要用文字把同样的信息讲清楚；如果是无人值守环境，不能假装有人会回答；如果是执行子会话，可以自动选推荐项但要报告。这一段体现了 `spec` 的核心设计：规格生成前，先把关键不确定性变成可回答的决策。

第四段是输出纪律。材料要求问题、选项、建议和完整性都必须足够具体；选项过多时要拆分，不能为了工具限制丢掉真实选项；涉及中文或其他非 ASCII 字符时要直接写原文，避免编码损坏；破坏性决定要比普通选择更强确认。这些规则服务一个目标：避免 agent 在“看似继续推进”的过程中偷偷替用户做产品决策。

第五段是后续工作连接。材料把 `spec` 放进 gstack 的更大循环里：从需求整理到计划审查、执行、测试、审查、发布、上下文保存和恢复。`spec` 不是终点，而是把混乱想法变成后续流程能够消费的输入。它还要求完成后报告做了什么、做了哪些决定、还不确定什么。

这份材料的关键约束包括：不在无人值守场景中等待不存在的人工回答；不把工具调用失败当作自动同意；不在超过工具选项上限时丢弃候选；不把一次性环境配置误当作业务规格；不把破坏性操作藏在普通确认里；不把宿主环境的偶发限制写成通用产品逻辑。

它的失败条件主要来自三类风险。第一类是决策失真：用户没有真正选择，但 agent 继续执行。第二类是范围污染：`spec` 被拿去做排查、发布、代码审查或同步配置，导致需求规格和执行动作混在一起。第三类是外壳污染：升级、遥测、路由、同步、会话探测等运行时内容挤占读者对核心方法的理解。

它的输出要求不是单纯生成文档，而是形成 backlog-ready issue 和本地 archive：要说明上下文、当前状态、 proposed change、验收标准、测试计划、回滚计划、工作量、相关文件、明确不做的范围；对于 audit/cleanup 类型，还要列完整 inventory、保留不应改动的部分和按风险排序的执行计划。对复杂选择，还要保留全部候选并通过分组或逐项提问完成筛选。

这份材料最值得提炼的设计亮点有三个。

第一，它把“提问”设计成产品能力，而不是临时交互。每个问题都必须携带背景、风险、推荐和取舍，让用户能做真正的决定。

第二，它区分了主体工作流和运行时外壳。主体是把模糊需求变成可执行规格；外壳是升级、遥测、路由、同步和宿主兼容。外壳能提升工具可用性，但不能抢走正文主线。

第三，它把自动化限制在有边界的场景里。执行子会话可以自动选推荐项，无人值守环境必须阻断，交互环境可以用文字 fallback。这个设计承认 agent 很强，但不允许它在关键产品判断上伪造用户同意。
