# 组织设计：真正要设计的是现实穿透公司的路径

这一章不要读成“组织要扁平一点”这么简单。它真正关心的是：当问题出现时，现实能不能尽快抵达那个真正能修正问题的人。

如果问题必须先从员工传给经理，再传给总监、副总裁，绕过好几层以后才到另一个真正执行的人手里，组织看上去仍然有秩序，现实却已经被稀释了。现场细节会丢失，坏消息会被修饰，责任会被包装成部门理由。最后这些组织问题不会只停留在会议室里，它会变成产品里的多余外壳、接口、重量、成本和流程。

本章里“盒中盒”的例子很关键。两个团队各自看都合理：一个团队想保护自己的部件，另一个团队也想完成自己的封闭结构。但产品合在一起，就出现了外壳套外壳。问题不一定是某个人愚蠢，而是组织边界让每个人都在局部正确地回答错误问题。产品里的冗余，常常是在提醒你：真实问题被部门边界挡住了。

所以，设计、工程和制造不能离得太远。制造现场最早遇到摩擦，设计端最有能力改动源头；如果两者之间隔着层层转述，组织就会把最有价值的反馈变慢。那句“手放在炉上会立刻抽回”的比喻，讲的不是情绪，而是反馈距离。谁最早感到痛，谁就最早知道哪里错。

“算法之道”也应该放在这个逻辑里看。先质疑要求，再删除，再简化或优化，然后才是加速和自动化。这个顺序不是工程审美，而是在防止组织把错误问题越做越漂亮。电池组玻璃纤维垫的例子说明，真正的改进有时不是让机器人更快，而是发现那个部件本来就不该存在。

这章值得吸收的，是把组织看成现实反馈系统：缩短问题到修正者之间的路径，减少局部正确造成的整体错误，先删除不该存在的要求，再谈效率。

需要降级使用的是直接沟通、失败容忍和快速试错。它们不是让管理消失，也不是让个人任性拍板；它们必须留在安全、成本、责任和专业边界之内。

必须拒绝的读法，是把这章读成“更狠、更快、更强控制”。真正有价值的不是英雄式施压，而是少让组织遮挡现实，少优化错误问题，少让恐惧阻断反馈。
