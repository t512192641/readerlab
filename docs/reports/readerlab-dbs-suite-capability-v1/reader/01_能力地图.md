# dbs-suite 能力地图 v1

## 先看结论

`dbs-suite` 不是 24 个零散工具的清单。它更像一套商业和内容工作的路由系统：先判断你现在的问题属于哪一类，再进入一个边界清楚、产物可验收的能力。

第一次不要按目录逐个读。先把它理解成 8 个能力域：入口路由、商业诊断、目标/问题/行动/学习、内容表达、发布包装、状态接续、内容资产工程、多角色/工作台治理。

## 第一次怎么读

1. 先读入口路由：`dbs` 只回答“现在该用哪个能力”，不替下游做诊断。
2. 再读诊断域：商业问题、概念、对标、快慢方法都先判断问题是否成立。
3. 然后读内容域：内容诊断、共鸣、传播、开头、AI 特征分别处理不同层面。
4. 最后读状态和工程域：保存、恢复、报告、决策系统、内容系统、聊天室、Agent 迁移都不是普通咨询，而是把结果接续、工程化或治理起来。

## 能力域地图

| 能力域 | 什么时候用 | 不要误用成 | 产物 | 验收点 | 包含 Skills |
|---|---|---|---|---|---|
| 入口路由与任务后导航 | 不知道该用哪个 dbs 能力，或做完一个诊断后问下一步 | 万能商业顾问 | 下游能力选择或 2-3 个下一步 | 没有越权诊断；路线能解释理由 | `dbs` |
| 商业诊断、概念消解与对标/方法判断 | 商业模式、概念含混、对标选择、快慢方法判断 | 目标管理、内容包装、行动激励 | 问题消解、体检报告、概念拆解、对标分析、慢方法诊断 | 说明问题在哪一层成立或消失；对标/慢方法能落到动作和资产 | `dbs-diagnosis`, `dbs-deconstruct`, `dbs-benchmark`, `dbs-slowisfast` |
| 目标、好问题、行动阻力与连续学习 | 愿望空转、问题说不清、知道不做、想连续学习 | 商业模式诊断本身 | 可验收目标、问题说明书、执行阻力诊断、连续学习文章 | 目标能失败；问题有反馈入口；行动诊断不鸡血；学习接住上一轮反馈 | `dbs-goal`, `dbs-good-question`, `dbs-action`, `dbs-learning` |
| 内容诊断、共鸣、传播、开头与 AI 特征 | 已有选题或文稿，需要判断内容机制、传播心理、开头或 AI 味 | 标题公式、排版、商业诊断 | 内容诊断、共鸣诊断、传播解码、开头方案、AI 特征报告 | 必须指向具体文本或选题；不能用技巧掩盖内容不成立 | `dbs-content`, `dbs-resonate`, `dbs-spread`, `dbs-hook`, `dbs-ai-check` |
| 标题公式与公众号发布包装 | 内容已基本成立，需要小红书标题或公众号 HTML | 内容质量判断 | 公式可追溯标题、公众号 HTML 和预览 | 标题能回到公式编号；HTML 不改观点 | `dbs-xhs-title`, `dbs-wechat-html` |
| 状态保存、恢复、报告与长期决策系统 | 已形成结论，需要保存、恢复、报告或长期跟踪 | 重新诊断、临场总结 | 存档、恢复摘要、报告、决策事件和状态文件 | 不存空壳；报告可追溯到存档；决策回填不抹掉历史 | `dbs-save`, `dbs-restore`, `dbs-report`, `dbs-decision` |
| 大量材料的内容资产工程 | 有大量本地素材，边界清楚，需要可重组内容系统 | 单篇文案整理、轻量素材归档 | 内容工程骨架、内容单元、主题地图、装配稿、处理状态 | 先审计再建工程；源素材不改写；样本通过后才批量 | `dbs-content-system` |
| 多角色讨论与 Agent 工作台迁移 | 需要多视角讨论，或治理 Claude/Codex/Grok 工作台 | 商业诊断、内容发布、安装教程 | 多角色讨论和判官总结；迁移审计、真源和 bridge 方案 | 聊天室要收束；迁移要分阶段确认和验证 | `dbs-chatroom`, `dbs-chatroom-austrian`, `dbs-agent-migration` |

## 真实任务怎么选

- 如果你只是说“帮我看看”“下一步怎么走”：先走 `dbs`。
- 如果你在问“这个业务/概念/对标/方法到底成不成立”：走商业诊断域。
- 如果诊断后暴露的是“目标不清、问题说不清、知道但不做”：走目标、好问题、行动或学习域。
- 如果已经有内容对象：先判断内容机制，再决定是否需要共鸣、传播、开头或 AI 特征检查。
- 如果内容已经成立，才进入标题公式或公众号发布。
- 如果这次结果需要下次接着用：走保存、恢复、报告或决策系统。
- 如果是大量本地素材，不是单篇内容：走内容资产工程。
- 如果是多视角讨论或 Agent 工作台治理：走聊天室或 Agent 迁移，不要混进商业诊断。

## 跨能力路线

| 路线 | 什么时候发生 | 关键判断 |
|---|---|---|
| 入口路由 -> 诊断 | 用户有具体商业问题、概念、对标或快慢方法疑问 | `/dbs` 只分发，诊断域才判断问题是否成立 |
| 诊断 -> 目标 / 行动 / 决策 / 好问题 | 诊断发现真正断点不是商业模式，而是目标、问题、执行或长期跟踪 | 不要让商业诊断吞掉所有后续动作 |
| 诊断 -> 内容表达 | 业务方向够清楚，瓶颈变成内容怎么做 | 内容域必须绑定已有选题或文稿 |
| 内容表达 -> 发布 | 内容基本成立，需要标题或公众号表面 | 标题和排版不能补救内容不成立 |
| 诊断 / 工作成果 -> save / report / restore | 已有结论、否决方向、待验证假设或下一步 | 状态工具只保存和合并已经形成的状态 |
| 大量材料 -> content-system | 有足够多素材、边界清楚、需要可重组内容工程 | 未过审计门槛时不进入重工程 |
| 聊天室 / Agent 迁移边界 | 多视角讨论或工作台治理 | 聊天室要主持人收束；迁移要真源和 bridge 验证 |

## 如何验收一次 dbs 输出

看四件事：

1. 它有没有守住自己的职责，没有替其他 Skill 做事。
2. 它有没有拿到必要输入，还是在信息不足时硬编。
3. 它的产物能不能按字段、公式、原文、诊断层级、存档或工程状态复核。
4. 它有没有在该转路由时转路由，而不是把所有问题都吞在自己里面。

如果这四件事都说不清，那就算文字写得顺，也只是一个像样的回答，不是合格的 dbs 能力输出。

## 读完后该记住什么

`dbs-suite` 的核心不是“很多工具”，而是“每个工具有边界，边界之间能路由，路由之后有产物，产物之后能验收和接续”。这也是 ReaderLab v1 这张能力地图和旧版目录清单的根本差别。
