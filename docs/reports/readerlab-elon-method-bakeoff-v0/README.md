# ReaderLab Elon Method Bakeoff v0

## 结论

ReaderLab v0 没有真正吸收好拆书/阅读类参考。它不是完全失败：来源契约、审计分离、全书结构和 reader-facing 边界比旧 demo 强；但它把仓颉、李继刚、乔木、book-to-skill 的方法拆成字段和 gate 后直接合成，缺少“各方法独立产出 -> 对照差异 -> 再重组”的中间验证。

因此当前判断是：ReaderLab v0 是一个可审查的工程样张，不是合格的深读/陪读产品。用户说“顶级食材炒成糊饭”的风险成立，证据如下。

## 实验输入

对同一本《埃隆之书》，让四种方法分别独立产出，不让它们改 ReaderLab v0：

- `cangjie-only.md`：按仓颉 RIA-TV++ 跑 Stage 0 / Stage 1 / Stage 1.5。
- `ljg-deepread-only.md`：按缺口、增量、机制、认知旅程、可复核结论做深读。
- `qiaomu-coread-only.md`：按章节共读、批注触发、误读纠偏做陪读。
- `book-to-skill-only.md`：按 trigger / input / steps / output / boundary 判断能否 Skill 化。

对照对象是 `docs/reports/readerlab-elon-full-product-v0/reader/*.md`。

## 对照证据

| 方法 | 独立产出了什么 | ReaderLab v0 有没有 | 判断 |
|---|---|---|---|
| 仓颉 | Stage 0 整书理解；5 类候选池；9 个候选的 V1/V2/V3；通过、降级、拒绝清单。 | 只有 6 张深读卡和 pending 验证字段，没有候选池和淘汰链。 | 字段化吸收，不是真吸收。 |
| 李继刚式深读 | 5 层理解缺口、增量/不增量、完整机制、7 步认知旅程、书内支撑/解释者推断/外部待验证分层。 | 有机制链，但像目录；没有缺口/增量/认知旅程/证据分层。 | 抓住骨架，但深读不够狠。 |
| 乔木共读 | 每部分读者真正该问什么；15 个批注触发点；正文附近误读纠偏；读者行动。 | 有阅读路线，但多是“先读什么再读什么”；没有批注触发点。 | 像报告，不像陪读。 |
| book-to-skill | 只保留 5 个可执行能力；每个都有 trigger、输入、步骤、输出、边界；淘汰 69 法则整体、硬核工作、多行星等。 | 深读卡混合了概念、案例、边界、术语、洞察；没有可调用能力结构。 | 对阅读可用，对 Skill 化不够。 |

## ReaderLab v0 的具体缺口

1. **没有候选池和淘汰机制**  
   仓颉输出里最有价值的是“哪些通过、哪些降级、哪些拒绝”。ReaderLab v0 只展示被选中的卡，没有说明为什么不是其他卡，也没有记录“硬核工作”“69 法则整体”等为什么不能升格。

2. **深读缺少增量判断**  
   李继刚式输出明确说：本书补的是“如何把宏大目标压成工程系统”的理解缺口，不补中立事实判断。ReaderLab v0 只说读者获得一个框架，没有说本书相对普通传记新增什么、没有新增什么。

3. **机制链还是太像目录**  
   ReaderLab v0 的“目标 -> 第一性原理 -> 工程 -> 组织 -> 制造 -> 案例 -> 文明边界”方向是对的，但没解释约束如何传导：为什么目标要求第一性原理，为什么第一性原理走向制造，为什么组织缺陷会变成产品缺陷。

4. **陪读场景没有真正出现**  
   乔木输出的核心是“读者读到这里会卡在哪里、该批注什么、怎么纠偏”。ReaderLab v0 读者页干净，但像压缩报告，缺少正文附近的提问和回应方向。

5. **Skill 化和阅读卡混在一起**  
   book-to-skill 输出保留 5 个真正可能成立的能力：第一性原理可行性拆解、算法之道流程瘦身、最短沟通路径诊断、交付系统瓶颈定位、单一核心指标反馈回路。ReaderLab v0 的卡片没有 trigger/input/steps/output/boundary，所以不能称为能力。

6. **“类型”标签对读者无意义**  
   `framework/principle/case/counterexample/term/transfer_insight` 是内部分类，不是读者收益。读者需要的是“我现在该怎么理解、怎么批注、怎么判断能否迁移”。

## 哪些东西应该保留

ReaderLab v0 的工程项仍有价值：

- `source-registry` / `location-map` / contracts 能保留审计边界。
- reader-facing 和 audit 分离是对的。
- 不把 36/36 spine coverage 说成精选译文正文完成是对的。
- 全书结构和第二部分重点选择有合理性。

但这些只是底座，不等于阅读质量成立。

## 重做原则

下一版不要继续修补 5 个短页面。应改成“一篇主阅读稿 + 审计附录”的形态：

1. **一篇主阅读稿**，合并现有 5 个 reader 页，删掉元说明和无意义类型标签。
2. **先写材料边界**：这是编校选编，不是连续传记或最终证据。
3. **用李继刚结构开篇**：这本书补什么缺口，新增什么，不新增什么。
4. **用仓颉结构承接**：候选池、通过/降级/拒绝、为什么。
5. **用乔木结构改写阅读路线**：每部分不是“先读什么”，而是“读到这里你该问什么”。
6. **用 book-to-skill 结构单独列能力**：只列 3-5 个可执行能力，其余降级为案例/术语/反例。
7. **保留审计 JSON**，但读者正文只给必要来源提示，不把内部字段露给读者。

## 下一版目标

`readerlab-elon-full-product-v1` 应该证明一件事：ReaderLab 不是把多个方法平均混合，而是按职责分层使用它们。

- 仓颉负责候选和筛选。
- 李继刚负责深读增量和机制。
- 乔木负责批注陪读。
- book-to-skill 负责判断哪些能变成能力。
- ReaderLab 负责把这些组织成可读、可批注、可复核的 LifeAtlas 阅读包。

如果 v1 不能清楚体现这五层职责，就不能说 ReaderLab 真正吸收了这些参考。
