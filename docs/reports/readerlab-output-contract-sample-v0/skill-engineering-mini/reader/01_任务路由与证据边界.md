# 任务路由与证据边界

这页展示三个防止 AI 过早下结论的基础模块：先判断该做什么，再确认能凭什么判断，最后检查产物是否真的适合读者。这里的 Agent 指执行阅读、整理或评估任务的 AI 工作单元；Skill 指一组可重复触发的能力或工作流。

## 材料正文（净化正文）

任务路由模块负责把模糊的用户请求分配到一个有边界的处理能力。它适用于用户只说出目标、材料或下一步愿望，但还没有明确要使用哪种 ReaderLab 能力的情况。这个模块先识别材料类型和当前目标，再选择一条下游路径；如果还不能唯一判断，就给出少量候选路径，并说明每条路径为什么适合、为什么其他近邻路径暂时不适合。它的关键约束是不能越权执行下游任务：路由只负责决定该走哪条能力链，不直接写深读结论，也不替代后续的证据读取或输出评估。比如用户只说“把这份材料做成可读包”时，路由应该先判断是书籍、长文还是工程材料，而不是直接生成一篇总结。

证据边界模块发生在任何解释和判断之前。它负责确认当前材料从哪里来、覆盖到什么范围、哪些位置可以支撑后续判断，以及哪些文件是主来源、哪些只是派生材料。它的用途是防止 AI 在证据不足时装作已经读完整份材料。覆盖范围可以只是样本、局部或完整材料；如果覆盖不足，只能支持目录地图、局部地图或结构诊断，不能生成完整全局理解。比如只读了一个模块样本，就只能说明这个样本里的结构和风险，不能说整个项目都采用同一套设计。

输出评估模块用于检查最终样本是否真的帮助读者阅读。它关注的不是文件是否齐全，而是读者页和审计材料有没有分开，正文有没有被 AI 摘要替代，关键判断有没有明确依据，技术洞察是不是只剩泛泛表扬，过度升格的结论有没有被压回合理范围，以及机器检查有没有被误说成人工验收。它的输出要求是给出可检查的评估结论，同时保留“仍需人工审查”的边界。比如测试通过只能说明规则检查通过，不能说明读者已经觉得这页好读。

这三个模块合在一起，形成了一个最小但完整的 Skill/工程材料阅读链条：先把请求送到正确能力，再确定解释能站在什么证据上，最后检查读者产物有没有越界。它们的设计亮点不是某个命令或文件格式，而是把 Agent 容易犯错的地方提前变成硬边界：不能没路由就动手，不能没证据就解释，不能有机器通过就宣布读者体验通过。

## 高阶讲解

这份材料真正讲的是“如何让一个会写东西的 Agent 先守住判断边界”。普通读者看到 Skill 或工程流程时，容易以为重点在目录、字段和检查清单；但这里更重要的是三个动作的先后关系。路由先回答“现在该做哪类事”，证据边界再回答“能凭什么做判断”，输出评估最后回答“这个结果是否真的适合读者”。如果顺序被打乱，系统就会出现看似勤快但实际危险的行为：一收到模糊请求就写结论，只读了局部样本就说全局规律，或者把工具检查通过说成读者已经认可。

对 ReaderLab 来说，这组三段材料是一个很小的样本，但它正好压中了复杂材料陪读的关键矛盾：读者需要的是可信正文和可理解解释，不是内部流程展示；而 Agent 需要的是足够明确的边界，避免把“我能生成”误当成“我已经理解”。因此，净化正文保留的是用途、触发条件、流程、约束、失败路径和输出要求，而不是执行外壳和内部证据。读者先读到这三个模块到底在做什么，再读 AI 的解释，才不会被审计语言牵着走。

从技术负责人视角看，这组设计的价值在于把“完成”的含义拆开了。路由完成不等于分析完成；证据登记完成不等于理解完整；机器评估通过不等于人工验收通过。这样的拆分会增加一点流程成本，但它能显著降低 Agent 在复杂材料中最常见的错误：过早下结论、过度升格、把内部状态包装成读者价值。

## 设计资产卡

### 卡片一：先路由，后执行

- 适用场景：用户给出模糊目标、复杂材料或“下一步怎么做”类请求，但还没有明确要进入哪种能力。
- 解决问题：防止 Agent 在没有判断材料类型和任务边界时直接生成下游内容。
- 原材料依据：任务路由模块明确要求先识别材料类型和当前目标，再选择下游路径，并说明近邻路径为什么不选。
- 可复用做法：把“选择能力”和“执行能力”拆成两步；路由输出只给推荐路径、候选路径和选择理由。
- 使用前提：系统里必须已经有可区分的下游能力，且用户请求中能提取出触发信号。
- 失败风险：如果路由理由太空，会变成形式化前置步骤；如果路由阶段偷偷执行下游任务，会破坏边界。典型反例是用户只问“这份材料怎么读”，系统却直接产出全局结论，后续才发现材料其实只覆盖了一个片段。
- 什么时候不要用：用户已经明确要求执行某个具体检查、修复或生成任务时，不要再额外制造路由层。

### 卡片二：证据不足时只给局部判断

- 适用场景：材料来源、覆盖范围或可引用位置还不清楚，但系统准备生成地图、解释或深读结论。
- 解决问题：防止局部样本被包装成完整理解。
- 原材料依据：证据边界模块要求先记录来源范围、覆盖状态和位置依据，并限制覆盖不足时只能输出目录地图、局部地图或结构诊断。
- 可复用做法：在解释前先声明“当前能支撑什么判断”，并把不能支撑的结论挡住。
- 使用前提：至少能识别主来源和派生材料，并能说明当前覆盖范围。
- 失败风险：如果只登记来源但不约束后续输出，证据层会退化成装饰。
- 什么时候不要用：没有可读来源时，不应生成局部判断；应该先阻塞并要求补足材料。

### 卡片三：机器通过不等于读者通过

- 适用场景：样章、阅读包或合同检查已经生成，需要判断是否能交给读者审查。
- 解决问题：防止文件齐全、字段通过或机器检查成功被误说成读者体验已经成立。
- 原材料依据：输出评估模块要求检查读者页和审计材料是否分离、正文是否被摘要替代、技术洞察是否具体，并保留人工审查边界。
- 可复用做法：把机器自检和人工验收分开命名；交付时只说当前证据能证明什么，不能证明什么。
- 使用前提：已经有读者页、审计材料和可检查的评估项。
- 失败风险：如果评估只看格式，不看阅读体验，就会再次把 contract 通过误当成产品通过。
- 什么时候不要用：内容还没有形成可阅读样张时，不要用输出评估替代实际生成。

## 必要提醒

这页只覆盖三个小模块，不能推断整个 ReaderLab 包已经完整可用。读者可以把它当成“工程材料如何变成可读正文”的样张，而不是把它当成完整产品说明。

这里的设计资产卡包含 Codex 的工程判断；它们适合帮助产品负责人复用方法，但不应被当成原作者逐字表达。
