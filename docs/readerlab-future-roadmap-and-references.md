# ReaderLab 后续功能与参考资料备忘

更新时间：2026-06-29  
用途：保存 v0.1 之后的功能、参考项目、设计线索和资料来源  
状态：项目内维护版；从 Downloads 原始备忘归档而来，不是 PRD

---

## 1. 当前已经确认的核心判断

### 1.1 ReaderLab 的长期方向

ReaderLab 可以长期扩展成一个“AI 陪读与知识沉淀系统”。

但第一版不能做重。2026-06-29 的 gstack、dbs-suite 和《埃隆之书》Demo 后，v0.1 的产品闭环可以视为基本跑通：ReaderLab 已能把复杂材料清洗、中文化、切段/分组，组织成 Obsidian 可读的 Markdown 阅读包，并验证真实 `tandem-comments` 批注链路。后续重点不应继续停留在“更多切段和翻译”，而应升级到全局理解和精华提炼。

当前路线从原来的“先 SkillReader v0.1，再扩展 ReaderLab”调整为：

```txt
v0.1 先跑通材料可读闭环
v0.4 / v0.5 优先建设全局理解层和精华提炼层
v0.6 再讨论升格到 Life Atlas 正式知识库或 Skill 草案
批注插件增强、HTML 展示、外部集成按实际积累后移
```

### 1.2 v0.1 的核心原则

- Obsidian 是阅读界面。
- Markdown 是数据本体。
- Codex 负责拆书、翻译、预处理、读取批注和回复。
- 批注优先采用现有 Obsidian 插件，不重新发明。
- 批注必须是就地批注，不是文末批注区。
- Life Atlas 先作为存放位置，不要自动污染正式知识库。
- 人类标注是核心，AI 不能替代人的第一次阅读判断。
- 自动导入阶段先服务“能读、能标、能问、能回复”；跨材料新知、导师式旁批、行业类比和思维模型连接先不进入默认生成器。

## 2. 通用链路备忘

长期可抽象为：

```txt
Capture → Normalize → Read → Annotate → Discuss → Distill → Promote
```

中文：

```txt
采集 → 统一格式 → 阅读 → 批注 → 讨论 → 提炼 → 升格
```

v0.1 已经基本跑通前五步中的最小闭环：

```txt
导入材料 → 拆成 Markdown 书 → 中文化 → Obsidian 批注 → Codex 回复
```

现在进入后两步之前的产品升级：先做“全局理解”和“精华提炼”，再决定哪些内容值得升格。

新的工作分层：

| 层级 | 目标 | 当前状态 |
|---|---|---|
| 原料处理层 | 清洗、中文化、切段、目录、正文保真 | v0.1 已基本通过 |
| 全局理解层 | 全书地图、能力地图、主线、概念关系、章节/Skill 功能关系 | 当前主线，优先建设 |
| 精华提炼层 | 框架、原则、案例、反例、术语、可迁移洞察 | 当前主线，参考仓颉/book-to-skill |
| 升格层 | 方法卡、概念卡、工作流、Skill 草案、LifeAtlas 正式沉淀 | 等 v0.4/v0.5 有积累后再做 |

## 3. 已看到的参考项目

### 3.1 qiaomu-read-helper

GitHub：  
https://github.com/joeseesun/qiaomu-read-helper

相关原始帖：  
https://x.com/vista8/status/2061118430305210492

核心信息：

- 面向深读一本书，而不是快速摘要。
- 把“章节文本 → 飞书批注 → AI 点评 → 个人读书笔记”做成可重复流程。
- 支持用户在飞书文档中加粗、划线、评论。
- AI 读取用户评论、加粗重点和上下文，再生成回复和读书笔记。

对 ReaderLab 的启发：

- 人先读、先标、先问，AI 再围绕痕迹回复。
- “人的阅读痕迹”比“AI 全文总结”更重要。
- v0.1 可以迁移它的方法论，但不采用飞书作为载体。

ReaderLab 取舍：

- 借鉴：章节化、批注驱动、AI 点评、个人化笔记。
- 暂不采用：飞书 CLI、飞书文档写入、完整读书笔记生成。

### 3.2 book-to-skill

GitHub：  
https://github.com/virgiliojr94/book-to-skill

核心信息：

- 把技术书、文档文件夹或资料集合转换成 Claude Code 可用的结构化 Skill。
- 支持从书和文档中抽取 frameworks、mental models、principles、techniques、anti-patterns 等。
- 强调输出不是一篇摘要，而是可被 agent 学习、引用、工作时调用的结构化 Skill。

对 ReaderLab 的启发：

- 大文本最终不只能变成笔记，也可以变成可调用知识资产。
- 长期可以增加“从阅读材料提炼 Skill 片段”的能力。
- 对读技术书、方法论书、教程资料很有参考价值。

ReaderLab 取舍：

- 借鉴：结构化、按需加载、从大文本中抽取可复用方法。
- 暂不采用：v0.1 不自动生成 Skill，不做完整 book-to-skill 流程。

### 3.3 gstack

GitHub：  
https://github.com/garrytan/gstack

Skills 文档：  
https://github.com/garrytan/gstack/blob/main/docs/skills.md

核心信息：

- 一套 Claude Code skill/workflow 包。
- 把开发过程拆成 CEO、Designer、Eng Manager、Release Manager、Doc Engineer、QA 等角色。
- 包含 `/spec`、`/review`、`/learn`、`/context-save`、`/context-restore`、`/guard` 等多种命令/Skill。
- 很适合学习复杂 Skills 如何组织、拆分角色、设计约束和串联流程。

对 ReaderLab 的启发：

- gstack 是第一批适合导入的测试材料。
- 重点不是学习“怎么用 gstack”，而是学习“它怎么写 Skills”。
- 可以用来测试 ReaderLab 是否能处理复杂 Skills 仓库。

ReaderLab 取舍：

- 借鉴：多角色拆分、阶段化工作流、Skill 写法、上下文保存/恢复、安全边界。
- 暂不采用：不照搬整套 gstack 到自己的 Codex 工作流。

### 3.4 Yao Open Skills / Yao WeRead Skill

GitHub：  
https://github.com/yaojingang/yao-open-skills

相关原始帖：  
https://x.com/yaojingang/status/2056019530829426912

核心信息：

- Yao Open Skills 是公开 Skill 合集。
- 方法线是 YAO = Yielding AI Outcomes。
- 强调不是继续堆 prompt，而是把方法、流程、评估、审美约束和执行边界沉淀成可复用 AI 资产。

对 ReaderLab 的启发：

- 长期需要分析“我的阅读痕迹”，不只是分析材料本身。
- 阅读系统可以有复盘和可视化，但这不是 v0.1。
- “把有效方法沉淀成可复用 AI 资产”与 Life Atlas / Skills 系统方向一致。

ReaderLab 取舍：

- 借鉴：阅读痕迹分析、长期复盘、AI 资产化。
- 暂不采用：微信读书接入、数据可视化、HTML 报告。

- 是否支持足够自然的正文旁批注。
- 是否方便 Codex 就地回复。
- 评论导出格式是否稳定、可读。

## 4. v0.1 之后的功能池

2026-06-29 之后，功能池不再按“全部暂不做”理解，而按优先级分层：

- 当前优先：全局理解层、精华提炼层。
- 中期后移：批注插件增强、阅读数据统计、HTML 展示。
- 需要积累后再做：Life Atlas 升格、自动生成正式 Skills。
- 仍保持按需：旁通视角、跨材料比较、外部搜索或 RAG。

### 4.1 旁通视角 / 跨材料新知

想法：

- 读者读到某段材料时，ReaderLab 不只重复材料结构，而是引入高质量外部理解资源。
- 这些资源可以来自其他行业案例、思维模型、已读书籍、LifeAtlas 旧笔记、已有 Skill、研究资料，或必要时的联网搜索。
- 目标体验类似“导师在旁边提供一个更高维视角”，或“微信读书旁边出现一条真正有质量的跨领域评论”。
- 它可以解释“这段材料在另一个领域里像什么”“它对应哪个思维模型”“它暴露了什么反例或风险”“它能迁移到哪里”。
- 这不是固定套用芒格、马斯克、六顶思维帽，也不是硬凑名人视角；具体引入谁或什么模型，应由材料、用户问题和可用知识源决定。

暂不做原因：

- 当前还没确定可靠来源。直接让生成器自由发挥，容易变成平庸 AI 解读或伪知识连接。
- 并不是每个 Skill、每个章节都值得做旁通视角；默认每页生成会增加噪音和成本。
- 这个能力可能更适合读者读完某段后主动提问时触发，而不是初次导入时自动生成。
- 如果后续实现，最好单独做成一个 Skill 或工具链，先读取材料上下文，再按需检索 LifeAtlas/思维模型库/已有读书经验/外部搜索。
- v0.1 先保留批注闭环：人先读、先标、先问，Codex 再围绕具体上下文回答。

可能进入 v0.4 之后，但不作为默认生成器页面。更合适的形态是：读后问答、批注触发、包级地图的待复核线索，或独立 Skill。

当前产品判断：

- 原来的“重点与亮点”不应继续承担这个职责。它把金句摘录、价值提炼和外部启发混在一起，导致三者都不稳定。
- 金句可以保留为“可划线句”：只忠于原文，数量可为 0。
- 高阶新知应另设“旁通视角”或类似模块：只有在有可靠来源、清楚触发点和明确读者收益时才生成。
- 初期可以用 `dbs-diagnosis` 做人工样张，但在样张通过前，不改生成器默认输出。

---

### 4.2 全局地图 / 能力地图

想法：

- 每个阅读包除了章节手册或主控清单，还应有包级全局理解页。
- 图书输出“全书地图”：核心问题、主线结构、部件功能、概念关系、方法/案例/反例地图、推荐阅读路线、待复核判断。
- Skill 包输出“能力地图”：能力域、入口 Skill、诊断/执行/表达/状态等功能关系、设计模式、可迁移方法、边界和风险。
- 它不是摘要。摘要压缩内容；全局地图解释结构、关系和可复用价值。

当前判断：

- 这是 v0.4 的主线能力。
- 可以先做手工/半手工 Demo，再回写生成机制。
- 必须标注置信度和来源范围，尤其是只读 TOC 或只完整处理部分章节时，不能假装已经读完整本书。

---

### 4.3 精华提炼 / 仓颉式蒸馏

想法：

- 参考仓颉 `book2skill` / book-to-skill 的方法，从材料中提取框架、原则、案例、反例、术语和候选 Skill 化方向。
- 对图书，重点看观点、论证、案例、模型和方法论。
- 对 Skill 包，重点看触发条件、输入输出、约束、工作流、失败条件、设计模式和可迁移能力。
- 提炼结果应是读后产物，不污染第一次阅读页。

当前判断：

- 这是 v0.5 的主线能力。
- 先做“候选提炼”，不直接生成正式 Skill。
- 候选必须保留来源、适用边界、置信度和待复核项。
- 可以产出方法卡、工作流草稿或 Skill 草案候选，但进入正式 LifeAtlas 或 canonical Skill 前必须人工确认。

---

### 4.4 模式索引

想法：

- 从多个材料中提炼共性模式。
- 例如：
  - 执行前澄清
  - 触发条件设计
  - 安全护栏
  - 上下文保存与恢复
  - 人类标注优先

暂不做原因：

- 容易变成大系统。
- 需要读过多个材料后才有价值。
- 应作为 v0.5 精华提炼后的跨材料积累，而不是早期自动生成长期原则。

---

### 4.5 Life Atlas 升格机制

想法：

- ReaderLab 是阅读实验区。
- 确认有长期价值的内容，再升格到 Life Atlas 正式区域。
- 可能去向：
  - `800_Skills与流程`
  - `300_方法论与卡片`
  - `600_开发能力库`
  - `700_产品与工具资产`

暂不做原因：

- 自动升格容易污染正式知识库。
- 需要人工确认。
- v0.1 先保留阅读记录，不进入正式库。v0.6 之后，当全局地图和精华提炼已有稳定样张，再讨论升格。

---

### 4.6 HTML 阅读界面

想法：

- Markdown 是数据本体。
- HTML 可以作为更舒服的阅读视图。
- 未来可做左侧目录、中间正文、右侧批注。

暂不做原因：

- Obsidian 已经能满足 v0.1 阅读和批注。
- 做 HTML 会增加工程量。
- 先验证 Markdown 体验是否足够。

---

### 4.7 PDF / EPUB / 网页导入

想法：

- 以后支持书籍、论文、网页、PDF、EPUB。
- 可以扩展 BookReader / PaperReader / DocReader。

暂不做原因：

- 材料解析复杂度高。
- v0.1 先处理 Skills 和 Markdown 文件。
- PDF / EPUB 应等插件和解析方案稳定后再接入。

---

### 4.8 Readwise / 微信读书 / 飞书接入

想法：

- Readwise / 微信读书可提供高亮和阅读痕迹。
- 飞书适合云文档协作。
- qiaomu-read-helper 和 yao-weread-skill 都提供了参考。

暂不做原因：

- v0.1 坚持本地优先。
- 先不要引入授权、API、云端依赖。
- 这些可以作为独立集成，不影响核心流程。

---

### 4.9 自动生成 Skills

想法：

- 阅读材料中发现的方法，可以沉淀成 Skill。
- book-to-skill 是强参考。

暂不做原因：

- 自动生成 Skill 风险较高。
- 需要人工判断是否真的可复用。
- v0.1 只做阅读，不做资产生产。v0.5 可以生成候选 Skill 化方向；正式 Skill 生成和安装应进入 v0.6 之后，并经过人工确认。

---

### 4.10 阅读数据统计和复盘

想法：

- 统计读了哪些材料。
- 哪些章节批注最多。
- 哪些主题反复出现。
- 生成周报/月报式复盘。

暂不做原因：

- 第一版没有足够数据。
- 容易偏离“先能读起来”的目标。

## 5. 未来可能的版本路线

### v0.1：单材料陪读闭环

目标：

- 书架
- 添加材料
- 拆书
- 中文化
- Obsidian 就地批注
- Codex 读取批注并回复

状态：产品闭环基本通过。gstack/dbs-suite 证明 Skills/Markdown 材料路径可行；《埃隆之书》Demo 证明书籍材料路径方向可行，但图书导入尚未机制化。

---

### v0.2：批注插件适配增强（后移）

目标：

- 确定主批注插件。
- 适配插件的真实 Markdown 存储格式。
- 支持 Codex 稳定识别批注、高亮、评论和回复位置。
- 支持“只处理新增批注”。

当前排序：

- 暂不作为下一阶段主线。
- 当前已有 `tandem-comments` 的最小真实链路和重刷保护；在没有更大批注量之前，继续深挖插件适配的收益不如全局理解/精华提炼高。
- 等积累更多真实阅读包和批注后，再做增强。

---

### v0.3：轻量 registry（基本够用）

目标：

- 记录已读材料。
- 记录材料类型、来源、状态、阅读目标。
- 简单关联本地已有 Skills。
- 不做复杂图谱。

当前排序：

- 已有 manifest、machine/human 状态、版本化 book id、批注保护和运行账本，足够支撑下一阶段测试。
- 先不单独建设复杂 registry。

---

### v0.4：全局理解层

目标：

- 为每个材料包生成包级“地图”，而不只是目录或主控清单。
- 图书：全书地图、核心问题、主线结构、部件功能图、概念/方法/案例关系、推荐阅读路线、待复核判断。
- Skill 包：能力地图、入口/诊断/执行/表达/状态等功能关系、设计模式、可迁移能力、边界和风险。
- 地图必须标注来源范围和置信度，不能在只处理部分章节时冒充全书定论。
- 旁通视角和跨材料比较暂时只作为地图中的候选线索，不进入默认页面。

---

### v0.5：精华提炼层

目标：

- 借鉴仓颉/book-to-skill，从阅读包中提炼框架、原则、案例、反例、术语和可迁移洞察。
- 生成“候选方法资产”，而不是默认生成正式 Skill。
- 区分材料类型：
  - Skills 看触发条件、输入输出、约束、流程、失败条件、设计模式。
  - 书籍看观点、论证、案例、模型、术语和反例。
  - 论文看问题、方法、实验、结论、局限。
- 输出可进入 v0.6 的候选：方法卡、概念卡、工作流、Skill 草案。

---

### v0.6：升格流程与候选 Skill 化

目标：

- 手动确认哪些内容值得进入正式知识库。
- 支持升格为：
  - 概念卡
  - 方法卡
  - 工作流
  - Skill 草案
  - 项目资料
- 只有经过 v0.4 地图和 v0.5 提炼验证的内容，才进入升格候选。
- 不自动污染 LifeAtlas `300/600/800`，不自动安装 Skill。

---

### v0.7：HTML 展示层

目标：

- 从 Markdown 生成 HTML 阅读界面。
- Markdown 仍是唯一数据本体。
- HTML 只作为展示层。

## 7. 当前最重要的产品约束

- 第一版必须能尽快用起来。
- 不要先做复杂 PRD。
- 不要凭空设计已有工具能解决的问题。
- 优先采用现有 Obsidian 插件。
- 不新增 LifeAtlas 顶层目录。
- 不自动污染正式知识库。
- 不让 AI 替代人的主动阅读。
- 不把阅读结果写成长篇总结垃圾。
- 一切以“能读、能标、能让 Codex 回答”为主。
