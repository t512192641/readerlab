# ReaderLab 阶段复盘

## 阶段结论

这次问题不是“章节循环方法已经失败”，而是主控 agent 把多阶段任务错误压缩成局部阶段判断，提前讨论 final boss 和交付条件，导致任务边界混乱。

当前真实进度：

- 《埃隆之书》正文级章节总数：15。
- 已按新方法通过章节循环：2 章，`成功之道`、`打造卓越团队`。
- 未开始：13 章。
- 全书总结：未启动。
- 仓颉 / 李继刚 / book-to-skill / 乔木 baseline 横向对比：未启动。
- 可迁移方法论 / Skill 草案：未启动。

正确初步交付不是“章节都有评价结果”，也不是“两个章节通过”，而是：

```text
15 个正文级章节全部 pass
-> ReaderLab 自己的全书总结 pass
-> 其他 Skills / 方法基线总结完成或纳入
-> 读者评价 agent 横向对比
-> 主控提炼可迁移方法论 / Skill 草案
-> 验证通过
```

## 真正解决了什么

- 明确了章节阶段的唯一队列：`docs/reports/readerlab-elon-chapter-loop-v0/chapter-queue.md`。
- 把当前覆盖事实固定为 `2 pass / 13 not_started`，避免说成“全书跑完后只有两章通过”。
- 把 final boss baseline 的名称、对照输出和本地/研究来源写清楚，避免只写“几个 Skills”却不给路径。
- 把下一会话启动入口压到 2 个文件：`AGENTS.md` 和 `docs/current-task.md`；其余只按 `current-task.md` 指令在执行时读取。
- 明确 baseline 只在章节全部 pass 且 ReaderLab 自己全书总结 pass 后启用，不能污染章节写作 agent。

## 项目开发问题

### 1. 把阶段完成误当成任务完成

现象：只跑了两个章节，却开始讨论 final boss、全书总结或方法论交付。

根本原因：没有把用户给出的完整任务链固化成硬门槛；主控把当前阶段的边界当成了全任务边界。

防重检查点：

- 每次汇报前先回答：当前完成的是哪一段链路？
- 如果后面还有 `->`，不得说“本轮交付完成”。
- 章节阶段只允许输出“章节阶段进度”，不能输出“全书能力成立”。

### 2. 停止标准错误

现象：把“所有章节有评价结果”或“跑过一些章节”当成可停止条件。

根本原因：没有区分 `pass`、`partial`、`fail`、`not_started` 对阶段推进的不同含义。

防重检查点：

- 章节阶段推进条件只能是所有正文级章节 `pass`。
- `partial/fail` 会阻断进入全书总结。
- 五轮仍未 `pass` 时，停止向后推进并记录 failure report，不强行落地。

### 3. 用护栏替代主问题

现象：把 “baseline shield 不够硬” 当作主问题，偏离了用户质疑的“为什么只跑两章就停”。

根本原因：把一个防污染约束当成了任务失败根因；没有正面回答实际完成了什么、按什么标准停下。

防重检查点：

- 用户问“为什么停止”时，必须先列事实进度和停止标准，不能先讲护栏。
- 护栏只能解释风险控制，不能解释任务完成。

### 4. 最小真相层膨胀

现象：把 15 个文件写成“最小真相层”。

根本原因：把可选背景、历史规格和当前执行必需事实混在一起。

防重检查点：

- 启动入口只保留治理规则和当前执行事实源。
- 队列、方法、验收、产品规格、包结构、ledger、research-log 都是按需扩展，不进入默认启动层。

### 5. baseline 名称没有路径

现象：写了仓颉、李继刚、book-to-skill、乔木等基线，但没有给具体文件、包路径或可调用状态。

根本原因：把“知道有这些方法”当成了“可执行输入”，没有补齐来源和状态。

防重检查点：

- 所有 baseline 必须写三项：名称、已有对照输出、本地/研究来源。
- 本地不可直接调用的 Skill 必须明确写成“未确认可调用”，不能暗示已安装可用。

### 6. 旧目标残留影响当前判断

现象：最早 `/goal` 文件仍是“只做一个小样章”，但当前会话已经被用户升级成整本《埃隆之书》闭环。

根本原因：旧目标文件被当作强事实，而最新用户指令没有被提升为当前控制范围。

防重检查点：

- 旧 `/goal` 和旧 handoff 只能做历史背景；最新显式用户指令控制当前范围。
- 如果旧目标与新指令冲突，必须在状态文档里写明“旧目标已过期”。

## 使用侧问题

### 1. 用户不得不重复纠正同一个范围问题

用户连续指出：只跑一个或两个章节不能进入最终挑战池；任务包含章节、全书总结、baseline 横评和方法论提炼。主控没有第一次就把完整链路写成硬门槛，增加了用户沟通成本。

### 2. 汇报没有直接回答“完成了什么”

主控一度没有用最小事实回答：跑了几个章节、哪些通过、哪些未开始、以什么标准停下。这让用户无法判断当前进度。

### 3. 术语和路径不够落地

“几个 Skills”“最终挑战池”“baseline”这些说法没有同步给出文件路径和本地状态，导致用户需要追问“去哪找”。

### 4. 文档过重，反而降低可恢复性

15 个启动文件不符合“最小真相层”的目的。它让下一会话更可能读到历史噪音，而不是直接执行下一章。

### 5. 主控没有及时承认方向错误

用户指出不匹配后，主控应该先承认错误的停止标准，再重写执行合同；不应继续解释局部护栏。

## 项目内规则候选

这些候选适合进入 `decisions.md` 或后续 `AGENTS.md`，不直接写成全局规则：

- 多阶段闭环必须有全链路完成定义；任何阶段通过都不能替代全任务完成。
- 阶段队列必须有分母、状态词和下一项；不能用“跑过”替代 `pass`。
- baseline / Skill 名称必须绑定路径和可调用状态。
- 启动入口默认只有 `AGENTS.md` 和 `docs/current-task.md`；更多文件必须由 `current-task.md` 明确说明何时读取。
- 新增方法规则必须标注 `method rule`、`book observation` 或 `rejected one-off fix`，防止过拟合。

## 可升维到全局错题本的候选

以下只作为候选；未获用户批准前不得写入 `~/.codex/global-knowledge/common-mistakes.md`。

### 候选 G-004：多阶段任务不能用局部阶段完成冒充全链路完成

- 域：Agent 流程 / 状态 / 记忆
- 现象：复杂任务包含多段链路，但 agent 完成其中一段后开始汇报整体交付或进入后续挑战。
- 根本原因：没有把完整链路写成硬门槛；阶段 gate 和最终 completion 混用。
- 正确做法：每次汇报都声明当前处于哪一段、完成了哪一段、下一段何时可进入。
- 防重检查点：看到 `A -> B -> C` 任务链时，任何 `A` 的完成都不能被表述成整个任务完成。
- 来源：ReaderLab / 2026-07-01 / 《埃隆之书》章节循环误停。

### 候选 G-005：最小上下文不能用文件数量堆安全感

- 域：Agent 流程 / 状态 / 记忆
- 现象：把大量规格、历史和背景文件列入“最小真相层”，导致启动成本高、历史噪音大。
- 根本原因：没有区分执行必需事实和按需背景。
- 正确做法：最小层只保留能直接恢复当前动作的文件；其余按需读取。
- 防重检查点：如果启动入口超过 `AGENTS.md` 和 `docs/current-task.md`，必须重新解释为什么不能由 `current-task.md` 按需路由。
- 来源：ReaderLab / 2026-07-01 / next-session prompt 修正。

### 候选 G-006：引用 Skill / baseline 时必须给出路径和可用状态

- 域：Agent 流程 / 状态 / 记忆
- 现象：方案里写了多个工具或方法名，但没有说明本地位置、输出文件或是否可调用。
- 根本原因：把概念引用当成执行资源。
- 正确做法：每个外部方法都写名称、artifact 路径、本地包路径或“未确认可调用”。
- 防重检查点：凡是出现“几个 Skills / 若干 baseline / 外部方法”，检查是否有路径和状态。
- 来源：ReaderLab / 2026-07-01 / final boss baseline 澄清。

### 候选 G-007：防护规则不能替代对主问题的回答

- 域：Agent 流程 / 状态 / 记忆
- 现象：用户问为什么任务停止，agent 回答某个 guardrail 不够硬，而不是回答实际进度和停止标准。
- 根本原因：把风险控制解释成任务解释；没有正面回答用户的问题。
- 正确做法：先答事实、标准和错误，再谈防护规则。
- 防重检查点：用户问“你完成了什么 / 为什么停”时，第一段必须是进度事实和停止标准。
- 来源：ReaderLab / 2026-07-01 / baseline shield 误答。
