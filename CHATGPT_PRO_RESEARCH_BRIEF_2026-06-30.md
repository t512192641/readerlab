# ReaderLab 给 ChatGPT Pro 的二次方案设计 Brief

## 一句话背景

ReaderLab 是一个复杂材料吸收系统：把书籍、长文、课程资料、代码文档、Skill 包和混合型资料转成正文优先的陪读包。它不是摘要器，也不是完整图书生成器。当前目标是让读者先读到可信的一手正文或净化正文，再得到真正有认知增量的 AI 高阶陪读。

当前项目已经从“怎么生成文件”推进到一个更难的问题：

```text
正文契约基本清楚了，但 AI 高阶陪读还没有可执行的方法论。
```

## 当前已确认的产品契约

ReaderLab 当前形态是：

```text
一手正文轨 + AI 高阶讲解轨 + 技术负责人 / 设计资产轨 + audit 事实层
```

### 书籍 / 长文

- 原样正文和章节顺序是主体。
- 默认不压缩、不改写、不重述。
- 只允许空格、空行、断行等轻量排版清理。
- AI 讲解只能在正文之后或旁边，不能抢正文位置。
- 没有原样正文的一手轨，reader experience 必须失败。

### Skill / 工程材料

- 净化正文是主体，不是 AI 摘要。
- 剥离安装命令、重复模板、执行外壳、机器状态、路径、hash、调试信息。
- 保留用途、触发条件、用户意图、核心流程、约束、失败条件、输出要求和设计亮点。
- 被剥离内容进入 audit、附录或技术负责人解析，不直接丢弃。

### Audit 分离

- source refs、location map、claim trace、machine/human 状态、降级/拒绝记录进入 audit。
- reader-facing 页不能把内部审计结构当主内容。

## 当前真实进展

### 已做的 real-source demo v1

当前 demo 目录：

```text
docs/reports/readerlab-real-source-demo-v1/
```

覆盖四个真实来源：

- `dbs-good-question`：真实 DB Skill。
- `dbs-decision`：真实 DB Skill。
- 《埃隆之书》“成功之道”：完整章节正文。
- 《埃隆之书》“打造卓越团队”：完整章节正文。

这轮验证说明：

- 书籍章节正文可以按完整章节放在 reader 页前面。
- Skill 源正文可以去掉 YAML/frontmatter 后保留主体，不再转译成 AI 说明。
- reader 页可以和 audit 分离。
- 这降低了“AI 导读替代正文”的风险。

但这不等于 ReaderLab 体验通过。

## 当前失败点

用户对《埃隆之书》两章 demo 的反馈很明确：

```text
正文契约方向是对的，但高阶讲解不够爽，不够通透。
它像普通书籍总结，不像一个见识丰富的人在高屋建瓴地陪读。
甚至不如过去参考过的某些 Skill 提炼方案。
```

当前高阶讲解的实际形态是：

```text
本章讲了什么
+ 一点防误读
+ 一点迁移建议
```

这只能算普通总结，不是 ReaderLab 想要的 AI 辅助阅读价值。

## 用户真正想要的体验

用户的表述大意是：

ReaderLab 的 AI 陪读应该既讲清重点和要点，又能高屋建瓴、旁征博引，用更大的价值观和体系重新看待材料。

它可以像一个拥有全局视角、见识丰富的人在陪读：

- 能讲清正文重点。
- 能指出局部材料背后的更大问题结构。
- 能引入跨学科思维模型、历史观、产品经验、组织管理、工程系统、商业案例、心理机制或用户既往经历。
- 能把看似不同学科和场景中的同构逻辑串起来。
- 能告诉读者哪些观点值得吸收，哪些只能降级，哪些必须拒绝。

但这些例子不是封闭清单。用户明确补充：这些只是其当下能力范围内能想到的表达。真正目标是更好的陪读服务，而不是机械套“芒格 / 历史观 / 跨学科模型”。

## 关键产品矛盾

我们现在面临的不是“需求不清”，而是“需求还没有转成操作层方法论”。

需要解决的矛盾是：

```text
AI 既要受正文约束，又要提供超出正文摘要的认知增量。
```

如果太保守，就会变成普通总结。

如果太发散，就会变成堆模型、堆名人、堆历史例子，脱离正文。

如果用固定栏目，就会像报告，不像陪读。

如果不设内部流程，Agent 每次都只能凭感觉写。

## 当前候选思路：高阶陪读生成法

我们初步设想过一个内部六步法，但还没有验证：

```text
1. 正文锚定
2. 普通读者会怎么读
3. 真正值得升维的问题
4. 高维镜头选择
5. 吸收 / 降级 / 拒绝
6. 写成自然陪读讲解
```

### 1. 正文锚定

先确认这一章原文到底能支持什么，不能先发散。

### 2. 普通读者会怎么读

模拟普通总结会停在哪里，用来识别真正要超越的层次。

### 3. 真正值得升维的问题

把“本章讲什么”改写成更大的问题。

例如《打造卓越团队》不只是“如何招优秀人才”，而可能是：

```text
一个组织怎样把人才、目标、反馈、原型和结果标准变成同一个现实校正系统？
```

### 4. 高维镜头选择

按材料选择最能照亮正文的镜头，而不是固定套模型。

候选镜头可能包括：

- 系统论 / 控制论。
- 历史组织形态。
- 产品组织理论。
- 经济学和激励。
- 心理学和反馈。
- 工程系统和瓶颈。
- 管理伦理和权力边界。

### 5. 吸收 / 降级 / 拒绝

对强观点做裁决：

- 吸收：能跨场景成立，且有正文支撑。
- 降级：有启发，但只适合特定条件。
- 拒绝：冲击力强，但容易误用，不能当方法。

### 6. 写成自然陪读讲解

最终 reader-facing 不展示上述流程，而是写成自然、通透、有判断力的讲解。

## 希望 ChatGPT Pro 帮忙的问题

请重点帮助我们设计“操作层方法论”，而不是直接写一个大而全产品功能清单。

### 1. 高阶陪读的底层模型应该是什么？

如何把“讲清重点 + 高屋建瓴 + 旁征博引 + 更大价值观和体系”拆成 Agent 可执行的内部步骤？

### 2. 如何避免两种失败？

失败 A：

```text
太保守，只是普通总结。
```

失败 B：

```text
太发散，堆模型、堆名人、堆历史例子，脱离正文。
```

### 3. Agent 应该如何选择“高维镜头”？

不是固定套跨学科模型，而是根据材料选择最能照亮正文的视角。

请设计一个镜头选择机制：

- 什么时候该用历史视角？
- 什么时候该用系统论 / 控制论？
- 什么时候该用经济学或激励？
- 什么时候该用组织理论？
- 什么时候该用心理学？
- 什么时候应该什么外部模型都不用，只讲材料内部机制？

### 4. 如何把“吸收 / 降级 / 拒绝”做成稳定 gate？

读者最需要的不只是“这点有道理”，而是知道：

- 什么值得学。
- 什么只在特殊条件下成立。
- 什么很有冲击力但不能照搬。

请设计这套判断的标准和输出形态。

### 5. Reader-facing 应该怎么写？

内部可以有步骤、候选池和 gate，但读者页不应像报告。

请设计最终讲解的表达形态：

- 应该是一段长讲解，还是几个自然小节？
- 是否需要标题？
- 如何兼顾“爽”和“严谨”？
- 如何避免固定栏目墙？

### 6. 如何验收高阶讲解？

请给一套可执行 eval：

- 重点是否讲清。
- 是否有普通总结没有的认知增量。
- 旁征博引是否真正照亮正文。
- 是否受正文约束。
- 是否区分原文、AI 解释、外部类比和待验证判断。
- 是否明确可吸收、降级、拒绝的内容。

### 7. 最小验证实验应该怎么设计？

我们倾向选择《埃隆之书》的“打造卓越团队”这一章，因为它涉及团队、反馈、人才、组织、产品和管理伦理，适合测试高阶讲解。

请设计一个最小实验：

- 输入材料是什么。
- Agent 内部步骤是什么。
- 输出产物是什么。
- 如何和当前普通总结版对照。
- 如何判断是否值得机制化。

## 不希望 ChatGPT Pro 做什么

- 不要建议先做完整自动图书生成器。
- 不要建议堆 UI、数据库、知识图谱或批注系统。
- 不要只给抽象理念，必须落到 Agent 可执行步骤。
- 不要把“跨学科”当成固定模型清单。
- 不要把高阶讲解写成不受正文约束的宏大散文。
- 不要把当前 demo 说成 product ready。
